IBM Sterling OMS 9.3 → Orchestrator-Based Microservices

Transition + End-State Architecture for Order Fulfillment Pipeline Modernization

IBM Sterling OMS 9.3 Retail / Ecommerce GCC PCI-DSS · SOC2 Kafka · Temporal · K8s Generated:
01

Executive Summary

IBM Sterling OMS 9.3 is a battle-tested, monolithic Order Management System that models order fulfillment as a Task Queue (TQ) pipeline — a series of sequenced tasks driven by a central agent polling model. While reliable at moderate scale, the architecture imposes critical constraints on modern retail demands: limited horizontal scalability, monolithic deployment coupling, SOAP-heavy APIs, proprietary JEE packaging (smcfs.ear), and no native event streaming capability. This guide details the complete migration from Sterling's Task Queue model to an orchestrator-based microservices architecture that preserves business logic continuity while delivering cloud-native scalability, observability, and independent deployability.

🎯 Architectural Decision
Temporal.io is selected as the orchestration engine over Apache Airflow, AWS Step Functions, or custom Saga coordinators. Temporal's durable execution model maps 1:1 to Sterling's task queue semantics — each Sterling task becomes a Temporal Activity, each fulfillment pipeline becomes a Workflow — while adding retry policies, timeouts, and compensation logic natively.

Key Scale Targets

  • 50K orders/day peak (flash sale: 5x = 250K)
  • P99 order creation < 500ms
  • P99 fulfillment workflow < 2s end-to-end trigger
  • 99.99% availability for payment + order capture
  • RTO < 15min, RPO < 1min for Order Service

Primary Constraints

  • Zero order data loss during transition
  • Sterling must remain live throughout migration
  • New + legacy systems must process same order corpus
  • PCI-DSS compliance for payment flows
  • GCC data residency (India DPDP Act compliance)

Sterling OMS Task Queue Model — As-Is Analysis

IBM Sterling OMS models fulfillment as a pipeline of Tasks. Each task corresponds to a business step and is managed by the Agent Server, which polls the YFS_TASK_Q table at configurable intervals. Tasks are picked up by agents, executed via Java-based User Exits or APIs, and transition the order to the next status. This is fundamentally an active polling / task dequeue pattern — not event-driven.

⚠ Sterling Limitations
The polling-based agent model creates ceiling bottlenecks at scale. DB contention on YFS_TASK_Q under high order volumes, monolithic JEE deployment (smcfs.ear), SOAP/XML API surface, and lack of native event streaming are primary modernization drivers.

Orchestrator Component Design

FulfillmentOrchestrator Workflow — Code Contract

// Temporal Workflow Definition — FulfillmentOrchestrator.java
@WorkflowInterface
public interface FulfillmentWorkflow {
    @WorkflowMethod
    FulfillmentResult execute(FulfillmentRequest request);
}

@WorkflowImpl
public class FulfillmentWorkflowImpl implements FulfillmentWorkflow {

    // Activity stubs with independent retry policies
    private final AddressValidationActivity addressActivity =
        Workflow.newActivityStub(AddressValidationActivity.class,
            ActivityOptions.newBuilder()
                .setTaskQueue("address-validation-queue")
                .setStartToCloseTimeout(Duration.ofSeconds(10))
                .setRetryOptions(RetryOptions.newBuilder()
                    .setMaximumAttempts(3)
                    .setInitialInterval(Duration.ofSeconds(1))
                    .build())
                .build());

    private final FraudValidationActivity fraudActivity = /* similar stub */ ...;
    private final PaymentConfirmationActivity paymentActivity = /* similar stub */ ...;
    // ... stubs for schedule, release, shipment, invoice

    @Override
    public FulfillmentResult execute(FulfillmentRequest req) {
        // Step 1: Address Validation
        AddressResult addr = addressActivity.validate(req.getShippingAddress());
        Workflow.upsertSearchAttributes(Map.of("OrderStatus", "ADDRESS_VALIDATED"));

        // Step 2: Fraud Check (saga — compensations registered)
        FraudResult fraud = fraudActivity.check(req.getOrderId(), req.getCustomerId());
        if (fraud.isHighRisk()) {
            // Compensation: no payment taken yet, just mark and exit
            return FulfillmentResult.rejected("FRAUD_HIGH_RISK", req.getOrderId());
        }

        // Step 3: Payment (compensate on downstream failure)
        PaymentResult payment = paymentActivity.authorize(req.getOrderId(), req.getAmount());

        try {
            // Steps 4-7: Schedule → Release → Shipment → Invoice
            scheduleActivity.schedule(req.getOrderId(), addr.getNormalizedAddress());
            releaseActivity.releaseToWMS(req.getOrderId());
            ShipmentResult shipment = shipmentActivity.confirmShipment(req.getOrderId());
            invoiceActivity.generate(req.getOrderId(), payment.getTransactionId());

            return FulfillmentResult.success(req.getOrderId(), shipment.getTrackingNumber());
        } catch (ActivityFailure e) {
            // Compensate: void payment auth
            paymentActivity.voidAuthorization(payment.getAuthCode());
            throw e; // Workflow fails, can be re-run from checkpoint
        }
    }
}

Transition Architecture — Sterling + Microservices in Parallel

The transition architecture runs IBM Sterling OMS and the new Microservices stack in parallel, connected by a bidirectional Event Bridge built on Kafka. An Order Router (feature-flag driven) controls what percentage of order traffic goes to Sterling vs. the new stack. Both systems share a consistent view of order state via the Event Bridge.

⚠ Dual-Write Consistency Risk
During parallel run, both systems write to their own databases. The Event Bridge synchronizer reconciles state asynchronously — this introduces a ~2-5 second eventual consistency window. Order status reads must route to the authoritative system for each order. Implement a routing table: ORDER_ID → SYSTEM (STERLING | NEW_MSA) in Redis.

    source_system   VARCHAR(20) NOT NULL,
    payload         JSONB,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()

Architecture Decision Records & Trade-offs

ADR-001 — Temporal.io as Orchestration Engine over Custom Saga Coordinator Accepted +

The 8-step fulfillment pipeline requires durable, retryable execution with compensation logic on failure. Sterling's Task Queue model has identical semantics. We needed an engine that maps naturally to TQ concepts without reimplementing orchestration infrastructure.

Adopt Temporal.io as the workflow orchestration engine. Each Sterling Task becomes a Temporal Activity. Each pipeline becomes a Workflow. Workers replace Agent Servers. This achieves 1:1 conceptual parity while gaining cloud-native durability, distributed tracing, and code-first workflow definition.

Pros: Durable execution (survives pod restarts), native retry/compensation, excellent observability via Temporal Web UI, polyglot SDK (Java/Go/Python). Cons: Operational complexity of Temporal cluster; team learning curve; Cassandra/Postgres persistence needs sizing.

AWS Step Functions: Vendor lock-in, 1-year max execution, JSON-only state machines — poor DX for complex compensation logic. Custom Saga Coordinator: Build cost, operational burden, re-inventing Temporal. Choreography-only (Kafka): No centralized visibility, distributed compensation extremely complex at 8 steps.

ADR-002 — Kafka as Event Backbone over IBM MQ Retain Strategy Accepted +

IBM MQ is Sterling's existing async integration backbone. Options were: retain MQ for new services, replace with Kafka, or run both. Kafka was evaluated against MQ for the new microservices stack.

Kafka becomes the event backbone for the new MSA stack. IBM MQ is retained only as Sterling's internal channel, bridged to Kafka via Kafka Connect MQ Source/Sink connectors during the transition period. MQ is sunset when Sterling is decommissioned.

Pros: Kafka enables log replay (critical for shadow validation), horizontal scaling, schema registry, consumer group isolation. Existing Kafka Connect MQ bridge is mature and production-ready. Cons: Two messaging systems during transition — operational overhead. Bridge introduces up to 500ms additional latency for cross-system events.

Retain IBM MQ for new services: No log replay, no schema registry, proprietary, limited community. Azure Service Bus / AWS SQS: Cloud lock-in, no log replay, limited throughput at scale.

ADR-003 — Strangler Fig + Feature Flag Router over Big Bang Migration Accepted +

The team evaluated two migration strategies: (1) Big Bang — build new stack, cut over on a date; (2) Strangler Fig — progressive traffic shift via feature flags with parallel run.

Strangler Fig with an Order Router (feature-flag service) and full shadow validation phase. The router directs configurable percentages of order traffic to Sterling vs New MSA. A Shadow Validator compares responses during dark launch to catch divergences before real traffic is shifted.

Pros: Zero-downtime migration, rollback in <1 min (flip flag to 0%), progressive confidence building, real production validation. Cons: Increased complexity of dual-system operation (12+ months); state synchronization overhead; engineers must understand both systems.

Big Bang: Unacceptable risk for order-critical system. Any defect = all orders affected simultaneously. No rollback except full revert. Blue-Green: Full parallel infrastructure cost; no partial traffic split capability.

ADR-004 — Orchestration (Temporal) over Pure Choreography (Kafka-only) Accepted +

Two patterns for coordinating the 8-step fulfillment pipeline: (1) Orchestration — central coordinator drives each step; (2) Choreography — each service reacts to events from the prior service via Kafka.

Orchestration via Temporal for the fulfillment pipeline. Kafka is used for cross-domain event publication (notifications, analytics, CDC sync) but the critical fulfillment path has a single orchestrator managing step sequencing and compensation.

Pros: Centralized visibility of workflow state, explicit compensation logic, easier debugging, matches Sterling TQ conceptual model. Cons: Orchestrator is a single point of logical failure (mitigated by Temporal's HA cluster). Services are less independently evolvable if orchestrator is tightly coupled to their APIs.

Pure Choreography: At 8 steps with compensation, choreography creates a "distributed spaghetti" of event dependencies. Debugging a failed order requires correlating 8 Kafka topics. Compensation is extremely difficult to implement correctly without a coordinator.
