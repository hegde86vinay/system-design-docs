DDoS & Inventory Hoarding Defense System Design Guide

Detecting and neutralizing bot-driven inventory reservation abuse in high-demand ecommerce checkout flows

Retail / Ecommerce Security / Anti-Bot PCI-DSS · GDPR Flash Sale Scale AWS · Cloudflare Generated:
01

Executive Summary

Problem: An ecommerce platform reserves inventory at checkout initiation — the moment a customer lands on the checkout page — rather than at confirmed payment. This design is common in high-demand limited-inventory scenarios (sneaker drops, concert tickets, flash sales, limited-edition product launches). The upfront reservation blocks real stock from other buyers for the reservation TTL window, typically 10–15 minutes. Bots and coordinated human actors exploit this by hammering the checkout page to hoard reservations without any intent to purchase, causing genuine customers to see "Out of Stock" while actual inventory sits locked in unconfirmed reservations.

⚠ Business Impact
This is not just a DDoS problem — it is a business integrity attack. Unlike a classic volumetric DDoS that aims to take the site down, inventory hoarding attacks are designed to stay under the radar while causing maximum commercial damage: suppressed conversion rates, customer trust erosion, and revenue loss that often isn't visible until post-event analysis.

Key constraints: Detection must operate at P99 < 50ms (inline with checkout flow), false-positive rate < 0.1% (wrongly blocking real customers is as damaging as letting bots through), reservation TTL is a hard business constraint, and the system must handle 50–200x traffic spikes during sale events with zero degradation in detection accuracy.

Primary architectural decision: A multi-layer defense-in-depth model — edge (Cloudflare WAF + Bot Management), gateway (behavioral fingerprinting + rate limiting), application (reservation circuit breaker + risk scoring), and async fraud pipeline (ML scoring + real-time feedback loop). No single layer is sufficient; the key insight is that bots evolved past single-layer defenses years ago.

Requirements

Functional Requirements

  • Detect and block bot-driven checkout page hits before inventory reservation is triggered
  • Classify incoming sessions as: legitimate human, suspicious, confirmed bot
  • Enforce per-user, per-device, per-IP reservation limits for high-demand SKUs
  • Implement reservation expiry with guaranteed release back to available pool
  • Support a virtual queue (waiting room) pattern for flash sale entry
  • Provide real-time inventory hoarding dashboards for ops teams
  • Support bot challenge escalation (CAPTCHA, device fingerprint challenge, 2FA)
  • Allowlist known good actors (loyalty members, pre-registered users)

Out of Scope

  • Payment fraud detection (separate system, post-payment)
  • Account takeover (ATO) — separate authentication concern
  • Content scraping defense
  • Full volumetric DDoS mitigation at L3/L4 — handled by CDN/cloud provider upstreams

Key Insight: The TTL Trap
If reservation TTL is 10 minutes and bots re-hit every 9 minutes, they can permanently lock inventory in a "zombie reservation" cycle. Even with TTL expiry, bot re-reservation can be faster than real customer discovery — you need to break the re-reservation cycle, not just rely on TTL cleanup.

Critical Design Decisions

💡 The "Gate Before Reserve" Principle
The single most impactful decision: no inventory touch until a session passes L1–L4 gates. Inventory reservation must be the last action, not the first. This seems obvious but is often violated in legacy OMS integrations where stock is reserved in a pre-checkout hook before auth/risk checks complete.

Detection Signals & Scoring

Signal Categories

🌐 Network Signals
  • IP reputation score (threat intel feeds)
  • ASN category (datacenter vs residential)
  • IP velocity: >N checkouts/min from same IP
  • Geo anomaly: IP geo ≠ shipping address
  • TLS fingerprint (JA3/JA4) mismatch vs declared UA
  • IPv6 /64 subnet concentration
  • Tor exit node / VPN detection
🖥 Device & Browser Signals
  • Canvas/WebGL fingerprint consistency
  • Navigator.webdriver = true (Selenium/Playwright)
  • Missing browser APIs (automation leak)
  • Font enumeration anomaly
  • Screen resolution / DPI outlier
  • Cookie/localStorage absence
  • Headless Chrome UA patterns
  • Mouse movement entropy (absent = bot)
⏱ Behavioral Signals
  • Time-to-checkout: < 2 sec from page load
  • No product page view before checkout
  • Form fill speed: too fast or perfectly spaced
  • Scroll/click events absent or synthetic
  • Session duration anomaly
  • Checkout-to-payment conversion: 0%
  • Repeated checkout for same SKU
👤 Identity / Account Signals
  • Account age < 24h + high-demand SKU checkout
  • No purchase history
  • Email domain: disposable (mailinator, etc.)
  • Multiple accounts sharing device fingerprint
  • Multiple accounts sharing payment method
  • Loyalty tier mismatch for early access
📦 Inventory Signals
  • Single user reserving across multiple SKU variants
  • Reservation-to-purchase ratio < 5%
  • TTL expiry + immediate re-reservation
  • Reservation spike correlated with sale start
  • SKU concentration: >X% of stock reserved by <Y% of users
🕸 Graph / Collusion Signals
  • Accounts sharing IP history
  • Accounts sharing device fingerprint
  • Accounts sharing shipping address
  • Account cluster activated within same 5-min window
  • Payment method sharing across accounts

  0–30   → ALLOW  — proceed to checkout

Key Detection: Time-to-Checkout Signal

⚡ Most Reliable Single Signal
A human browsing a high-demand product page, adding to cart, and initiating checkout takes typically 45–180 seconds from first page load. Any session hitting the checkout API within 2–5 seconds of session start — with no product page view event — is almost certainly a bot. This signal alone, combined with IP reputation, catches >80% of simple bot attacks.

Reservation Circuit Breaker

The reservation service implements a circuit breaker pattern specifically for hoarding scenarios. When the reservation rate for a SKU exceeds a defined threshold (e.g., more than 30% of stock reserved in under 60 seconds with <5% conversion), the circuit trips:

// Pseudocode: Reservation Circuit Breaker
class ReservationCircuitBreaker {

  STATES = { CLOSED, OPEN, HALF_OPEN }

  // Trip conditions (per SKU)
  tripConditions = {
    reservationVelocity: "> 30% stock reserved in < 60s",
    conversionRate: "< 5% reservation-to-payment in last 5min",
    uniqueIPConcentration: "> 50% reservations from < 10 IPs",
    botScoreAggregate: "> 40% sessions with risk score > 50"
  }

  // OPEN state actions (circuit tripped)
  openActions = [
    "Pause new reservations for SKU",
    "Flush zombie reservations (risk score > 70)",
    "Trigger CAPTCHA for all new sessions",
    "Notify ops team via PagerDuty",
    "Log to fraud audit trail"
  ]

  // Re-close after 2min if conditions normalize
  halfOpenCheck = "Allow 5% traffic through, measure conversion"
}

Anti-Re-Reservation: Cooldown Enforcement

After a reservation expires without purchase, enforce a per-user/device cooldown before allowing a new reservation for the same SKU. This breaks the bot re-reservation cycle:

// Redis-based cooldown after TTL expiry
SET reservation:cooldown:{userId}:{skuId} 1
EXPIREAT reservation:cooldown:{userId}:{skuId} {unix_ts + cooldown_seconds}

// Cooldown config by risk tier:
// CLEAN session expired  → 30 sec cooldown
// SUSPECT session expired → 5 min cooldown  
// HIGH-RISK expired      → 30 min cooldown + ops review

Virtual Waiting Room Architecture

For known high-demand SKU launches, replace immediate checkout access with a metered virtual queue. This caps the concurrent checkout load AND makes bot attacks less effective since queue position is randomly assigned rather than first-come-first-served:

// Virtual Queue Design (Redis-backed)

// On sale start:
// 1. Assign all arriving users a random queue position token
// 2. Admit N users per second based on checkout service capacity
// 3. Show real-time position estimate to queued users
// 4. Assign queue tokens with session binding (prevents resell/transfer)

Key Redis structures:
  ZADD queue:sku:{skuId} {timestamp + random_jitter} {sessionId}
  // Jitter prevents exact arrival-time ordering — removes bot advantage
  // of hitting exactly at T+0 vs 5ms later

Admission rate: 
  checkout_service_capacity / avg_checkout_duration_seconds
  Example: 200 concurrent checkouts / 45s avg = ~4 admissions/sec

Bot impact:
  - Bots can queue, but queue position is randomized
  - Queue token is bound to device fingerprint
  - Transferring queue tokens is detected by device mismatch
  - Bots holding queue positions without checkout conversion 
    are expelled after 30s at front of queue

ADRs & Trade-offs

ADR-001 — Gate Before Reserve: Risk scoring must complete before inventory touch Accepted +

In many existing OMS integrations (including IBM Sterling), inventory reservation is triggered early in the checkout pipeline — sometimes before authentication completes. Bots exploit this to reserve stock before any gate can inspect the request.

Enforce a strict "no inventory touch" rule until the session clears L1–L4 gates. Risk scoring is synchronous and inline. Inventory reservation is the final step, not the first.

Adds ~20–30ms latency to the checkout initiation path (risk scoring compute). Requires refactoring existing checkout flow that calls OMS early. Benefit: bots never touch inventory even if they reach the checkout API.

Post-reservation async cancellation only: too slow — bots can re-reserve faster than async pipeline cancels. Pre-payment reservation only: breaks UX since customers need to know stock is held for them during payment input.

ADR-002 — Randomized Queue Positioning (not FIFO) for high-demand SKU waiting rooms Accepted +

A strict FIFO waiting room rewards whoever hits the endpoint first — which is always bots. Bots with precise timing infrastructure will consistently win FIFO queues, defeating the purpose of metered entry.

Assign queue positions with a random jitter (e.g., timestamp + random(0, 30 seconds)) within a 60-second window at sale start. All users arriving in the first 60 seconds are treated as equal, and admission order is randomized among them.

Less deterministic for genuine users too — some real customers who arrived at T+0 will be behind others who arrived at T+30s. Transparent communication ("you're in the queue, positions are randomized to ensure fairness") can mitigate perceived unfairness. Neutralizes the bot timing advantage.

Strict FIFO: always won by bots. Lottery (random selection from all arrivals): acceptable but less transparent than queue metaphor. First-come-first-served limited to authenticated loyalty members only: good supplemental measure but leaves guest traffic unprotected.

ADR-003 — Shorter Adaptive TTL for high-risk sessions vs fixed 10-min TTL Accepted +

A fixed 10-minute reservation TTL is the standard UX design to give customers time to enter payment details. However, bots exploit this to keep inventory locked for 10 minutes per cycle, and simply re-hit at TTL-1 minute.

Implement risk-tiered TTLs: clean sessions get 10 minutes; suspect sessions get 3 minutes; the post-TTL cooldown for suspect sessions is 5 minutes (preventing immediate re-reservation). This is invisible to genuine users (who won't be in suspect tier) and dramatically reduces bot hoarding window.

Requires storing risk tier in reservation record. Requires risk score to be available at reservation time (already mandated by ADR-001). Edge case: a real user mis-classified as suspect gets only 3 minutes, which may be too short for payment entry. Mitigation: CAPTCHA pass bumps them to clean tier automatically.

Global TTL reduction to 3 minutes: hurts all users including genuine ones who need time to enter card details. No TTL (reservation held until explicit abandon): catastrophic for hoarding. Server-side heartbeat renewal: bots can send heartbeats too.

ADR-004 — Async ML pipeline for deep scoring (not inline) with async cancellation Accepted +

Deep ML scoring (graph analysis, historical patterns, collusion rings) takes 500ms–2s. Adding this to the inline checkout path would create an unacceptable latency hit.

Run fast heuristic scoring inline (target <50ms). Publish reservation event to Kafka. Async ML pipeline consumes events, scores deeply within 2 seconds, and triggers cancellation if high risk. The reservation is made but may be cancelled within 2 seconds — before the customer has a chance to enter payment.

Small window (~2s) where a bot has a valid reservation before ML cancels it. Acceptable trade-off given inline scoring already blocks the vast majority. If ML cancellation arrives while customer is entering payment, the checkout page shows "reservation expired — item no longer available" — same UX as TTL expiry. Requires idempotent cancellation handling in checkout service.

Fully synchronous ML scoring: adds 500ms–2s to checkout load, user-visible. No async ML at all: inline heuristics alone miss sophisticated attacks. Pre-reservation ML scoring (before any reservation): too slow to be practical for all sessions at peak load.

Interview Cheat Sheet — Director / Principal Level

🎯 Top 5 Clarifying Questions

  • When does reservation happen — on checkout page load, or on explicit user action? (This determines attack surface)
  • What's the reservation TTL, and is it business-negotiable? (Shorter TTL = smaller hoarding window)
  • Do we have authenticated vs guest checkout? (Guest checkout is massively harder to defend)
  • What's the acceptable false-positive rate — would we rather block 1% of real users or let 5% of bots through? (This calibrates the threshold strategy)
  • Are we defending against casual scalpers, organized bot networks, or both? (Drives sophistication of the ML pipeline)

⚖ Trade-offs to Verbalize Proactively

  • False positive rate vs false negative rate — explicitly state you're optimizing for low FP first in ecommerce (blocking real customers is a brand crisis)
  • Inline latency vs detection depth — async ML is the right answer; explain why
  • Build vs Buy for bot detection — acknowledge Cloudflare Bot Mgmt wins on residential proxy detection; in-house can't replicate their network view
  • Fixed TTL vs adaptive TTL — proactively raise this; it's a non-obvious improvement that interviewers love
  • Randomized queue vs FIFO — this is a "wow" insight most candidates miss

🔍 Failure Modes Interviewers Probe

  • "What if bots start using real residential IPs?" → Behavioral signals become primary; IP alone is insufficient — this is why you need the multi-signal composite score
  • "What if the ML pipeline is down?" → Defense still runs via heuristic L1–L4 layers; ML is additive, not critical path
  • "What if a real flash sale causes the same traffic pattern as a bot attack?" → Calibrate thresholds per-event; baseline changes for known sale windows (shift alert thresholds upward during planned events)
  • "How do you handle the ML model being gamed by bots learning your signals?" → Model retraining loop, feature drift detection, signal diversification
  • "What about human bot farms that pass all technical signals?" → Purchase limits per identity + loyalty gating for high-demand SKUs is the last resort

✨ "Wow" Depth Signals

  • Randomized queue positioning: Mentioning that FIFO queues always reward bots — and proposing randomized admission windows — signals deep domain thinking
  • Adaptive TTL by risk tier: Most candidates talk about TTL but don't differentiate by session risk — this is a nuanced, implementable insight
  • Circuit breaker on reservations: Applying the circuit breaker pattern not to service failures but to business metric anomalies (conversion rate) shows systems thinking beyond textbook definitions
  • JA3/JA4 TLS fingerprinting: Mentioning TLS fingerprinting at the edge demonstrates edge security depth beyond just IP blocking

❓ Questions to Ask the Interviewer

  • Has the team experienced a specific hoarding incident? What was the impact and what existing mitigations are in place?
  • Is the OMS (Sterling or custom) reservation flow decoupled enough to add a pre-reserve gate, or is it tightly coupled to the checkout pipeline today?
  • What's the current reservation-to-purchase rate for high-demand SKUs — do we have that baseline?

📌 Sterling OMS-Specific Context

  • Sterling's GIV (Global Inventory Visibility) and ATP (Available-to-Promise) calls are synchronous — wrapping them in a reservation circuit breaker requires a sidecar or proxy layer
  • Sterling's reservation can be implemented as a pending order with hold — these orders clutter OMS if not cleaned up; automated TTL-based cancellation pipelines are critical
  • The pipeline agent framework in Sterling is too slow for real-time risk gate integration — the gate must live upstream of OMS, not inside it
