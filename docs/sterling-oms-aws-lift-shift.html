Executive Summary

This guide covers a Rehost (Lift & Shift) migration of IBM Sterling Order Management System (OMS) version 9.3 — the traditional, non-containerized edition — from an on-premises data centre to AWS. The deployment preserves the original monolithic stack: IBM WebSphere Application Server ND 8.5.5 on EC2, the smcfs.ear application archive, Sterling Agents, IBM MQ (or Amazon MQ), and Oracle Database on Amazon RDS. No microservices are built. No containers are involved. The only structural change is that physical servers become EC2 instances and bare-metal Oracle becomes RDS Oracle.

⚠️ Important Context
IBM officially re-architected Sterling OMS at v10.0 to use certified containers on OpenShift. The v9.3 traditional deployment (smcfs.ear on WAS) is the pre-container era stack. IBM's preferred cloud path from v9.3 is to upgrade to v10.0+ certified containers — but many GCC teams must first lift-and-shift 9.3 to AWS as a stepping stone, to gain cloud operational maturity before undertaking re-platforming. This guide addresses that specific scenario.

What Changes vs. What Stays the Same

✅ What Stays the Same

  • smcfs.ear build process and deployment
  • Sterling Installation Manager-based install
  • WAS ND cell topology (Dmgr + Node Agents + Cluster)
  • Oracle DB schema, JDBC config, Sterling DB XML
  • Agent framework (JVM-based agents)
  • IBM MQ queue definitions and JMS bindings
  • customeroverrides.properties customizations
  • IHS (IBM HTTP Server) as web tier

🔄 What Changes on AWS

  • Physical servers → EC2 instances (RHEL 8.x on HVM)
  • On-prem Oracle → Amazon RDS for Oracle (EE or SE2)
  • Physical NAS/SAN → Amazon EFS (NFS mount)
  • Hardware load balancer → AWS ALB (Application Load Balancer)
  • DNS / hostname → Route 53 private hosted zones
  • VPN/MPLS → AWS VPC with Direct Connect or Site-to-Site VPN
  • Backup tapes → AWS Backup + S3 snapshots

Key Constraints

  • Zero new microservices: smcfs.ear deployed as-is onto WAS 8.5.5 on EC2
  • Oracle RDS: Oracle EE or SE2, matching the on-prem Oracle version (11gR2 or 12c)
  • IBM License continuity: WAS, MQ, Sterling OMS licenses must be transferred or BYOL on EC2
  • Availability SLA: 99.9% (3 nines) minimum with Multi-AZ RDS and EC2 Auto-recovery
  • RPO/RTO: RPO ≤ 1 hour, RTO ≤ 4 hours via RDS automated backups + AMI snapshots

Key Sterling 9.3 Concepts for AWS Migration

📋 Critical Configuration Files
  • customer_overrides.properties — Contains DB connection strings, MQ hostnames, port overrides. This file changes when migrating to AWS (RDS endpoint replaces on-prem Oracle hostname).
  • sandbox.cfg — Environment-specific properties including ANT_OPTS, DB vendor, JDBC driver class. Update Oracle JDBC URL to RDS endpoint.
  • jdbc.properties — Primary JDBC datasource. Replace host:port:sid with RDS endpoint.
  • WAS server.xml / resources.xml — WAS-level JDBC datasource definitions; must point to RDS endpoint post-migration.

EC2 Server Setup — WAS ND Installation

This section covers the step-by-step setup of the EC2 instances that will host the WAS ND cell for Sterling OMS 9.3.

Base OS Configuration (RHEL 8 on EC2)

# 1. Launch EC2 with RHEL 8 AMI (HVM, 64-bit)
#    ami-0b4c7755cdf0d9219 (RHEL 8.x ap-south-1 — check latest)

# 2. Mount EFS for Sterling shared directory
sudo mkdir -p /opt/IBM/Sterling
sudo mount -t nfs4 -o nfsvers=4.1,rsize=1048576,wsize=1048576,hard,timeo=600,retrans=2,noresvport \
  fs-xxxxxxxx.efs.ap-south-1.amazonaws.com:/ /opt/IBM/Sterling
# Add to /etc/fstab for persistence

# 3. OS-level prerequisites for WAS 8.5.5 on RHEL 8
sudo yum install -y \
  libstdc++ glibc compat-libstdc++-33 libXp libXmu \
  libXau libxcb libX11 libXext libXt libXtst \
  gtk2 libgcc java-1.8.0-ibm

# 4. Create sterling OS user (never run WAS as root)
sudo useradd -m -s /bin/bash sterling
sudo usermod -aG wheel sterling

# 5. Set kernel parameters (ulimits for WAS JVMs)
echo "sterling soft nofile 65536" >> /etc/security/limits.conf
echo "sterling hard nofile 65536" >> /etc/security/limits.conf
echo "sterling soft nproc 16384"  >> /etc/security/limits.conf

# 6. Disable SELinux (WAS 8.5 requirement)
sudo sed -i 's/^SELINUX=enforcing/SELINUX=permissive/' /etc/selinux/config

# 7. Set hostname (WAS cell uses hostname — must be consistent)
sudo hostnamectl set-hostname was-appsvr-1.sterling.internal

IBM WebSphere Application Server ND 8.5.5 Installation

# IBM Installation Manager (IM) — required to install WAS
# Download IM from IBM Passport Advantage and transfer to EC2 via S3

  -templatePath /opt/IBM/WebSphere/AppServer/profileTemplates/management \

# 4. Start Deployment Manager

# 5. Create managed node profile on App Server EC2s
/opt/IBM/WebSphere/AppServer/bin/manageprofiles.sh -create \

  -templatePath /opt/IBM/WebSphere/AppServer/profileTemplates/managed \
  -hostName was-appsvr-1.sterling.internal \

# 6. Federate managed node to Deployment Manager

Oracle Schema Migration Steps

  1. Export On-Prem Oracle Schema (Data Pump)

    Run Oracle Data Pump export on on-prem: expdp sterling_user/password DIRECTORY=DATA_PUMP_DIR DUMPFILE=sterling_schema.dmp LOGFILE=export.log SCHEMAS=STERLING_USER. Transfer dump file to S3.

  2. Create Sterling DB User on RDS

    Connect to RDS and create the Sterling schema user with same privileges as on-prem: CREATE USER sterling_user IDENTIFIED BY password DEFAULT TABLESPACE STERLING_DATA TEMPORARY TABLESPACE TEMP; GRANT CONNECT, RESOURCE, DBA TO sterling_user;

  3. Create Required Tablespaces

    Sterling OMS requires dedicated tablespaces. Create STERLING_DATA, STERLING_INDEX, STERLING_LOB tablespaces on RDS with appropriate sizes.

  4. Import Schema via Data Pump

    Copy dump file from S3 to EC2 bastion, then import: impdp sterling_user/password@rds-endpoint:1521/ORCL DIRECTORY=DATA_PUMP_DIR DUMPFILE=sterling_schema.dmp LOGFILE=import.log REMAP_SCHEMA=sterling_user:sterling_user

  5. Update Sterling JDBC Configuration

    Update customer_overrides.properties and sandbox.cfg on EFS to point to RDS endpoint. Replace: yfs.dbproperty.url=jdbc:oracle:thin:@rds-endpoint.amazonaws.com:1521:ORCL

  6. Run Sterling DB Installer (if schema changes needed)

    If applying patches: run DBXchanger or Sterling DB installer to apply any pending DB schema updates.

JDBC Configuration in customer_overrides.properties

# customer_overrides.properties — DB section (updated for AWS RDS)
# Location: $STERLING_HOME/properties/customer_overrides.properties

# Oracle JDBC URL for RDS (thin driver — no Oracle client needed)
yfs.dbproperty.url=jdbc:oracle:thin:@sterling-oms-prod.xxxx.ap-south-1.rds.amazonaws.com:1521/ORCL

yfs.dbproperty.driver=oracle.jdbc.OracleDriver
yfs.dbproperty.username=sterling_user

# Use Secrets Manager ARN or encrypted value — never plaintext in properties
# Best practice: inject from SSM Parameter Store at startup
yfs.dbproperty.password=

# Connection pool settings (WAS manages the pool, but these are for Sterling's own pool)
yfs.dbproperty.minPoolSize=5
yfs.dbproperty.maxPoolSize=50
yfs.dbproperty.connectionTimeout=30000
yfs.dbproperty.maxStatements=100
⚠️ RDS Oracle Limitations vs On-Prem Oracle
  • No SYS or SYSTEM access — use RDSADMIN for DBA tasks
  • No Oracle RAC on RDS — use Multi-AZ for HA (synchronous replication)
  • Data Pump directory must be on RDS-managed directory (DATA_PUMP_DIR via UTL_FILE)
  • Some Sterling DBA maintenance scripts (reorg, statistics) need adaptation for RDS stored procedures
  • No direct filesystem access — use S3 integration for exports/imports

Sterling OMS 9.3 Installation on EC2

Sterling OMS 9.3 uses IBM Installation Manager (IM) for installation. The recommended approach for lift-and-shift is to clone the existing on-prem installation rather than fresh-install, to preserve all customizations.

Approach A — Clone On-Prem Installation (Recommended)

# On-Prem: Create a compressed archive of the Sterling install directory
tar -czf sterling_install_backup.tar.gz \
  /opt/IBM/Sterling/ \
  /opt/IBM/WebSphere/ \
  /var/ibm/InstallationManager/

# Transfer to S3
aws s3 cp sterling_install_backup.tar.gz s3://your-sterling-artifacts/backups/

# On EC2: Pull from S3 and restore
aws s3 cp s3://your-sterling-artifacts/backups/sterling_install_backup.tar.gz /tmp/
tar -xzf /tmp/sterling_install_backup.tar.gz -C /
chown -R sterling:sterling /opt/IBM/

Approach B — Fresh Install with IBM Installation Manager

# 1. Stage Sterling 9.3 binary (from IBM Passport Advantage) to S3, copy to EC2
aws s3 cp s3://sterling-artifacts/sterling-93-linux.tar.gz /tmp/
tar -xzf /tmp/sterling-93-linux.tar.gz -C /tmp/sterling-repo

# 2. Install Sterling OMS 9.3 via IBM IM (silent mode)
/opt/IBM/InstallationManager/eclipse/tools/imcl install \
  com.ibm.sterling.multiorder.enterprise \

Zero-downtime DB migration with validation

  • Run Sterling DB validation scripts: row counts, constraint checks, index validity
  • Performance benchmark: run key Sterling queries against RDS vs on-prem Oracle
Phase 3 — Parallel Run & Testing (3 weeks)
Full parallel environment with functional + load testing
  • Build and deploy smcfs.ear to WAS cluster on EC2
  • Full regression: order create, payment, sourcing, fulfillment, returns flows
  • Load test: JMeter test simulating peak traffic against AWS environment
  • Agent testing: pipeline transitions, integration server connectivity
  • Validate all inbound/outbound integrations

Cutover Procedure (Maintenance Window)

  1. Stop incoming traffic

    Update external DNS or WAF rules to send 503 maintenance page. Quiesce on-prem IHS.

  2. Final Oracle Data Pump sync

    Export only changed data since last full export using Oracle Log Miner or incremental Data Pump. Import delta to RDS.

  3. Drain on-prem MQ queues

    Ensure all in-flight messages on on-prem IBM MQ are processed before switching. Wait for agent queue depths to reach zero.

  4. Validate AWS environment

    Smoke test: create a test order on the AWS environment. Confirm DB write, agent pickup, and pipeline transition.

  5. DNS cutover

    Update Route 53 (or corporate DNS) to point to AWS ALB. TTL should be pre-set to 60 seconds for this window.

  6. Monitor for 30 minutes

    Watch CloudWatch metrics: WAS JVM heap, RDS connections, MQ depth, order processing rates.

  7. Rollback trigger

    If error rate > 1% or order processing stops: revert DNS to on-prem, investigate. On-prem environment kept warm for 7-day rollback window.

Architecture Decision Records (ADRs)

ADR-001 — Rehost (Lift & Shift) over Re-platform to Containers Accepted +

The team must move Sterling OMS 9.3 to AWS. IBM's preferred path is to upgrade to v10.0 certified containers on OpenShift. However, re-platforming to containers requires significant effort: custom extensions must be rebuilt, CI/CD pipelines created, and team must learn Kubernetes/OpenShift. Timeline constraint: 3-month window.

Rehost as-is: run smcfs.ear on WAS ND on EC2, Oracle on RDS. Defer containerization to Phase 2 (post-stabilization on AWS). This preserves all existing customizations, WAS configuration, and agent setup without change.

Pros: Faster migration (weeks vs months), zero application code change risk, team uses existing WAS/Sterling skills. Cons: Does not leverage cloud-native scaling; WAS ND licensing cost continues; IBM v9.3 end-of-life support risk increases over time. This is a stepping stone not a permanent state.

Containerize to v10.0 immediately: Too risky in 3-month window — requires upgrading Sterling version, rebuilding EAR, learning Kubernetes. SaaS OMS: Rejected due to extensive customizations (200+ custom APIs) that don't translate to SaaS.

ADR-002 — Amazon RDS Oracle over Self-Managed Oracle on EC2 Accepted +

Sterling OMS 9.3 requires Oracle DB. The team can run Oracle on EC2 (self-managed) or use Amazon RDS for Oracle (managed service). The on-prem Oracle team has deep expertise but AWS cloud DBA skills are limited.

Use Amazon RDS for Oracle with BYOL (Bring Your Own License) and Multi-AZ. This offloads patching, backup, and HA to AWS while using the existing Oracle license.

Pros: Automated backups (PITR), Multi-AZ failover, automatic minor version patching, no need to manage Oracle RAC. Cons: No SYS/SYSTEM access — some DBA maintenance scripts need adaptation. No Oracle RAC equivalent. RDS parameter group limitations may require some query tuning.

Oracle on EC2: Team would need to set up RAC or active-passive, manage Oracle Clusterware, handle backups. Operational overhead too high. Oracle Autonomous Database: Not supported by Sterling OMS 9.3; Sterling requires traditional Oracle JDBC connectivity with schema creation privileges.

ADR-003 — IBM MQ on EC2 over Amazon MQ for JMS Accepted +

Sterling OMS 9.3 requires JMS messaging. On-prem uses IBM MQ 7.5 with custom queue manager config, MQ bindings file, and some IBM MQ-specific APIs. Amazon MQ (ActiveMQ) is available as a managed service.

Run IBM MQ 8.x on a dedicated EC2 instance (m5.large) using the existing queue manager configuration. The MQ config is migrated as-is, ensuring zero compatibility risk.

Pros: Zero JMS compatibility risk; same queue manager name, channel names, and bindings file; existing Sterling MQ integration code unchanged. Cons: Self-managed MQ; need to patch and operate IBM MQ on EC2; no managed HA (need to set up MQ active-passive manually or use RDQM).

Amazon MQ (ActiveMQ): Sterling 9.3 uses IBM MQ-specific com.ibm.mq.* classes in some integration adapters. Switching to ActiveMQ would require code changes and testing — violates the lift-and-shift constraint. Revisit in Phase 2 (containerization).

ADR-004 — Amazon EFS over EBS for Sterling Shared Directory Accepted +

Sterling OMS requires multiple EC2 instances (App Server 1, App Server 2, Agent Server) to share the same Sterling install directory, extensions, and configuration files. On-prem this was a NAS/SAN mount.

Use Amazon EFS for the shared /opt/IBM/Sterling/ mount. EFS provides NFS-compatible, multi-AZ shared storage accessible simultaneously from multiple EC2 instances.

Pros: Multi-AZ durable storage; automatic scaling; NFS mount is familiar to existing team; eliminates single-point-of-failure for shared config. Cons: EFS latency is higher than local EBS (1-3ms vs <0.5ms); for log-heavy workloads, EFS throughput may be a bottleneck (use Provisioned Throughput mode if needed).

EBS shared via instance: EBS volumes can only be attached to one instance at a time (except io1/io2 multi-attach, which is complex to configure correctly for file sharing). S3 via S3FS: POSIX compatibility issues with S3FS; Sterling's file operations require true NFS semantics.

Interview Cheat Sheet — Director/Principal Level

🎯 Top 5 Clarifying Questions to Ask First

  • What is the current Oracle version on-prem — 11gR2 or 12c? Does RDS support it in the same version or do we need to upgrade?
  • What are the existing WAS JVM heap settings? This determines EC2 instance sizing.
  • Are we using IBM MQ-specific APIs (com.ibm.mq.*) or pure JMS? This determines Amazon MQ vs IBM MQ on EC2 decision.
  • What are the inbound/outbound integrations — FTP, HTTP, JMS, MFT? These all need to route to AWS post-cutover.
  • What is IBM Sterling license model — per-CPU perpetual or subscription? Does moving to cloud affect the license?

⚡ Key Trade-offs to Verbalize Proactively

  • Lift & Shift preserves risk: All existing WAS performance issues, Oracle tuning debt, and agent design debt come to AWS intact.
  • RDS vs EC2 Oracle: RDS loses Oracle RAC but gains managed HA (Multi-AZ) — net positive for most GCCs.
  • IBM MQ licensing: Running IBM MQ on EC2 requires an IBM MQ license — don't underestimate this cost in TCO.
  • WAS ND EOL risk: WAS 8.5.5 extended support ends — lifting to AWS doesn't extend IBM support. The team must plan a v9.3 → v10.0 containerization roadmap.

🐛 Failure Modes Interviewers Probe For

  • Hostname mismatch: WAS cell uses hostnames; EC2 hostnames change on reboot unless properly configured with Route 53. Node federation breaks.
  • Oracle character set mismatch: If on-prem is WE8ISO8859P1 and RDS is AL32UTF8, Data Pump import fails on special characters.
  • open_cursors exhaustion: Sterling's heavy Oracle cursor usage hits default limits in RDS parameter groups. Tuning required.
  • EFS burst credits: EFS bursts to high throughput then throttles when burst credits are exhausted. Heavy log-writing agents can saturate EFS during peak.
  • MQ in-flight messages during cutover: Not draining MQ queues before DNS cutover results in duplicate or lost order processing.

✨ "Wow" Moments — Principal-Level Depth Signals

  • Dual-write DB during cutover: For near-zero downtime, use Oracle GoldenGate or AWS DMS to replicate on-prem Oracle → RDS in real-time, then cut DNS with seconds of downtime — not hours.
  • customer_overrides.properties → Secrets Manager: Don't just copy the config file to EFS; refactor startup to pull sensitive values (DB password, MQ password) from AWS Secrets Manager at JVM startup. This is a GCC security maturity signal.
  • Build server lifecycle: The smcfs.ear build server should be on a Spot Instance that terminates after each build — saves 70% cost and ensures clean build environment every time.

❓ Questions to Ask the Interviewer

  • What is the team's current experience level with AWS? Is there a platform engineering team, or is the Sterling team expected to own infrastructure?
  • Is the goal purely lift-and-shift, or is this a stepping stone to Sterling 10.0 containers? That changes the investment in infrastructure automation.
  • Is there a Direct Connect from on-prem to AWS, or will we need Site-to-Site VPN for the integration layer?

📋 Migration Checklist — What People Forget

  • JDBC Oracle driver version — on-prem may use ojdbc6; RDS Oracle 19c requires ojdbc8
  • WAS Admin Console SSL certificates — self-signed certs expire; renew before migration
  • Sterling Elasticsearch (if used) — must also be migrated or replaced with OpenSearch
  • Sterling Integration Server FTP/SFTP adapters — external FTP hosts must whitelist AWS Elastic IPs
  • NTP/timezone alignment — EC2 instances must use the same timezone as on-prem to avoid Sterling schedule agent drift
