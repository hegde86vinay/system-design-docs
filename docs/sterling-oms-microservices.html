Practitioner's guide for deconstructing IBM Sterling Order Management into cloud-native, domain-driven microservices for a retail enterprise

01

Executive Summary

Problem: IBM Sterling OMS 9.3 is a Java EE monolith running on WebSphere Application Server. For most retailers, it has become the epicenter of deployment bottlenecks, expensive IBM licensing, slow feature velocity, and brittle integrations with WMS, TMS, POS, and eCommerce. Scaling is all-or-nothing; a surge in BOPIS (Buy Online, Pick Up In-Store) traffic brings down fulfillment APIs too. The application's XML-heavy pipeline architecture and tight coupling to DB2 make horizontal scaling practically impossible.

⚠ Sterling Reality Check
Sterling's internal architecture — Agent Framework, time-triggered agents, condition trees, YCD/YFS API layers — is deeply stateful and difficult to decompose naively. This guide treats Sterling as an anti-corruption layer problem, not just a rewrite. The wrong approach is a big-bang rewrite. The right approach is strangler-fig with aggressive event sourcing.

Key Constraints & Scale Targets

Scale Targets

  • Order ingestion: 5,000–50,000 orders/hour (peak sale events 10x)
  • Order status API: P99 < 150ms
  • Fulfillment decisions: P99 < 500ms (SLA-sensitive)
  • Availability: 99.95% for order capture; 99.9% for fulfillment
  • RPO: 0 (zero data loss on order records)
  • RTO: < 15 minutes for order capture path

Constraints

  • IBM license cost: eliminate or reduce to minimal Sterling footprint
  • Zero-downtime migration — orders must never be lost
  • Multi-channel: eCommerce, POS, marketplace, call center
  • Existing WMS / 3PL integrations must be preserved during transition
  • Audit trail requirement: every order state transition logged (PCI + internal audit)
  • India DPDP Act compliance for customer PII

Primary Architectural Decision

Decompose Sterling using Strangler Fig + Event-Driven Saga Pattern. Sterling remains authoritative for active in-flight orders during transition, while new microservices capture all new order traffic. A Kafka-based event backbone replaces Sterling's internal agent-to-agent messaging. CQRS is applied to Order Management and Inventory to handle mixed read/write traffic patterns. The Sterling DB2 schema is incrementally migrated domain-by-domain using CDC (Debezium).

Customer PII (name, phone, address) in order records falls under India's Digital Personal Data Protection Act. All microservices handling customer data must implement data minimisation, purpose limitation, and deletion pipelines. Do not replicate raw PII into Kafka topics — use tokenisation or reference IDs in events.

High-Level Architecture

📌 Current State Context
Sterling OMS 9.3 uses IBM MQ as its primary async ingestion backbone. Order capture channels (eCommerce, POS, marketplaces) drop order XML messages onto dedicated MQ queues. The createOrder agent (a time-triggered or continuous JMS listener) picks these up, validates, and calls the createOrder YFS API internally. Direct synchronous API calls to Sterling's HTTP endpoint exist but are a secondary path — primarily used by call centre integrations and some internal systems.

Key As-Is Characteristics to Preserve

IBM MQ Async Ingestion — Why It Exists

  • Decoupling: Channel systems (ATG, POS) are decoupled from Sterling availability — MQ acts as a durable buffer during Sterling restarts
  • Backpressure: The createOrder agent controls processing rate; channels can burst-drop messages without overwhelming Sterling's pipeline
  • Channel normalisation: Each channel adapter transforms its native format into Sterling XML before dropping to MQ — Sterling sees a single canonical schema
  • DLQ handling: Failed messages go to ORDER.DLQ for ops triage — this is your current incident management hook

Agents as Async Processors — Migration Targets

  • createOrder Agent → Order Capture Service (Kafka consumer replacing JMS listener)
  • Payment Collection Agent → Payment Service (triggered by Kafka saga event)
  • Sourcing Agent → Fulfillment Sourcing Service (Drools engine)
  • Release Agent → WMS Adapter Service (Kafka consumer → WMS API)
  • Alert Agent → Notification Service (Kafka consumer → Twilio/SES)
  • Note: Each agent that has a timer schedule must be mapped to a cron job or Temporal scheduled workflow — do not assume they're purely event-driven
⛔ Critical Discovery: Agent Inventory Before Phase 2
Run this query on your Sterling DB2 to list all deployed agents before writing a line of new code: 
SELECT AGENTNAME, AGENTCRITERIA, FREQUENCY, FREQUENCYINMINUTES,
       NEXTSCHEDULEDEXECUTION, ISRUNNING
FROM   YFS_AGENT_CRITERIA
ORDER  BY FREQUENCY, AGENTNAME;
Every row here is a piece of business logic that needs a new-stack equivalent. Typical retailers have 30–60 active agents, many undocumented.
🔀 Transition Architecture — Dual Path (Phase 1–3)
IBM MQ is retained as the ingestion backbone during transition. A new MQ-to-Kafka bridge service translates IBM MQ messages into Kafka events. The Sterling createOrder agent continues to process legacy channel traffic. New channels are onboarded directly to Kafka. Feature flags at the API Gateway control which path each channel uses. This preserves channel uptime — no channel system needs to change during Phase 1–2.

MQ → Kafka Bridge: Design Detail

The MQ-to-Kafka Bridge is the most critical new service in Phase 1. It must be deployed before any channel migration begins. It acts as a translation and normalization layer:

  • Input: Subscribes to IBM MQ queue ORDER.CREATE as a JMS MessageListener (Spring JMS or Camel)
  • Transform: Parses Sterling XML order payload → validates against canonical Avro schema → enriches with sourceChannel, ingestTimestamp, mqMessageId (for deduplication)
  • Output: Publishes to Kafka topic order.intake with order ID as partition key (ensures ordering per order)
  • DLQ bridge: IBM MQ ORDER.DLQ messages are also bridged to Kafka topic order.intake.dlq for unified ops monitoring
  • Idempotency: MQ message ID tracked in Redis (TTL=24h) to prevent duplicates on MQ redelivery

💡 Why Not Replace IBM MQ Immediately?
Channel systems (ATG, POS software, marketplace adapters) are tightly coupled to IBM MQ — they use MQ client libraries, queue names, and message formats that took years to stabilize. Changing them requires separate projects for each channel. The bridge approach means zero changes to channel systems during the OMS migration. IBM MQ can be decommissioned as a separate initiative once all channels are migrated to direct Kafka producers (Phase 4+).
✅ Target State — Full MSA (Post Phase 4)
IBM MQ is fully decommissioned. All channel systems publish directly to Kafka (or via the API Gateway for synchronous channels). Sterling OMS and WebSphere are sunset. The order lifecycle is driven entirely by Kafka events + Temporal sagas. Both sync (REST → API GW) and async (direct Kafka produce) ingestion paths are supported natively.

🔑 Dual Ingestion: Sync vs Async in Target State

The target architecture intentionally supports both ingestion patterns — the async MQ pattern that channel systems relied on is preserved in spirit, just with Kafka replacing IBM MQ:

Sync Path (API Gateway)
  • eCommerce web checkout (user waiting)
  • Call centre (agent needs immediate order ID)
  • B2B API callers expecting sync ACK
  • Returns 202 Accepted + orderId immediately
  • Order Capture Svc handles via REST endpoint
Async Path (Direct Kafka)
  • POS terminals (fire-and-confirm pattern)
  • Marketplace batch feeds (Flipkart, Amazon)
  • High-volume EDI order drops
  • Producer gets Kafka ACK (offset committed)
  • Order Capture Svc handles via consumer group

Detailed Component Design

Order Capture Service

Responsibility

The direct equivalent of Sterling's createOrder Agent + YFS createOrder API — but built for dual ingestion. It handles two entry points: a REST endpoint for synchronous channel callers (via API Gateway) and a Kafka consumer for async channel callers (via the MQ→Kafka bridge or direct producers). Both paths converge on the same core order validation, inventory soft-hold, and persistence logic. The service is the single authority for assigning an orderId and transitioning the order to PENDING state.

Sync Path (REST — replaces CC direct API call to Sterling)
POST /v1/orders
{
  "channel": "WEB|CALLCENTER|B2B",
  "customerId": "C-UUID",
  "items": [{ "skuId": "...", "qty": 2, "price": 1299.00 }],
  "fulfillmentType": "SHIP|BOPIS|SDD",
  "shippingAddressRef": "ADDR-UUID",
  "paymentTokenRef": "PAY-UUID"
}

→ 202 Accepted (immediate):
{
  "orderId": "ORD-2026-XXXXXX",
  "status": "PENDING",
  "estimatedConfirmation": "PT30S"
}
Async Path (Kafka consumer — replaces MQ → createOrder Agent)
Consumer group: order-capture-svc
Topic: order.intake
Partition key: channel + session_id
                (ensures ordering per cart)

On consume:
1. Deserialise Avro → canonical order
2. Validate schema + business rules
3. Soft-hold inventory
4. INSERT order (PENDING) + outbox
5. Commit Kafka offset
   (only after DB write succeeds)
6. Status available via GET /v1/orders/{id}

Failure Modes & Mitigation

  • Sync path — inventory reserve fails: Return 409 with stockout reason; no Kafka event published
  • Async path — consumer lag > threshold: HPA triggers more pods; alert if lag > 5,000 messages on order.intake
  • Both paths — DB write fails after inventory hold: Outbox pattern — order row + outbox row written atomically; hold released on rollback
  • Async path — poison message (unparseable): Message sent to order.intake.dlq; ops alerted; does NOT block queue (unlike IBM MQ DLQ which could stall agents)
  • Kafka unavailable (async path): MQ bridge buffers messages on IBM MQ — inherent durability preserved; no data loss
  • Sync path — Kafka unavailable for downstream: Outbox relay queues events; sync caller already has 202 + orderId; no customer impact

Scaling Strategy

Stateless; horizontal scale on EKS behind HPA triggered at 70% CPU or Kafka consumer lag > 2,000. Target 50 pods during peak sale. REST endpoint behind Kong rate-limiting (10k req/min burst). Kafka consumer parallelism = number of order.intake partitions (set to 60 for peak throughput). Connection pool to PostgreSQL via PgBouncer (transaction mode, max 200 connections).

Order Orchestrator (Saga Engine)

Responsibility

Implements the Choreography + Orchestration hybrid Saga for order lifecycle. Replaces Sterling's Pipeline/Condition Tree framework. Each order state transition is an event. The orchestrator holds the saga state machine and drives compensating transactions on failure.

Saga Compensation Chain

  • payment.failed → publish inventory.release → mark order CANCELLED
  • sourcing.failed → publish payment.void → publish inventory.release
  • wms.exception → pause saga, alert ops team, await manual intervention event

Fulfillment Sourcing Service

Responsibility

Implements the inventory sourcing / fulfillment optimization logic that was Sterling's most business-critical and hardest-to-replicate component. Routes orders to the optimal fulfillment node (DC, store, 3PL) based on inventory availability, proximity, cost, SLA, and store capacity. Replaces Sterling's Sourcing Rules Engine.

Key Design Decisions

  • Rules engine: Use Drools / OpenL Tablets for declarative sourcing rules — business teams can configure without code deployment
  • Inventory data: Read from Inventory Service (Redis cache) — must tolerate ~30s staleness for performance
  • Response SLA: 500ms P99 (synchronous); async fallback publishes result to Kafka within 2s
  • Multi-node splits: Supports split shipments; publishes one fulfillment.node.assigned per shipment group

Inventory Service

Responsibility

Real-time available-to-promise (ATP) inventory. Replaces Sterling's YFS Inventory module. Three layers: physical inventory (from WMS via CDC), reserved inventory (soft-holds during order capture), and available inventory = physical − reserved − safety stock.

Data Model Sketch

inventory_node {
  sku_id, node_id, physical_qty,
  reserved_qty, safety_stock_qty,
  atp_qty (computed),
  last_wms_sync_ts, version
}

soft_hold {
  hold_id, sku_id, node_id, qty,
  order_id, expires_at, status
}

Scaling Strategy

Write path: PostgreSQL with optimistic locking on version column. Read path: Redis with TTL=30s. Cache invalidated on every WMS CDC event. During flash sales, Redis read ratio is >95% — this absorbs the spike without hitting Postgres.

💡 Key Difference from Sterling's createOrder Agent
In Sterling, the createOrder Agent was a single-threaded (or lightly multi-threaded) JMS listener that processed messages sequentially per queue. The new Order Capture Svc Kafka consumer uses partition-level parallelism — 60 partitions = 60 concurrent processing threads, each independently offset-committed. This eliminates the head-of-line blocking problem that caused Sterling agent backlogs during peak events.

Migration Strategy

⛔ Anti-Pattern Warning
Do NOT attempt a big-bang rewrite of Sterling. Retailers who have tried this have faced 18-month delays and catastrophic order loss events. Sterling's pipeline logic encodes years of edge cases (partial shipments, split tender, international addresses, marketplace-specific SLAs). These must be discovered, not assumed.

Phase 0 — Discovery & Baseline (8 weeks)

  • Sterling API audit: Log every YFS API call for 30 days. Classify by domain, volume, caller system.
  • Pipeline mapping: Export all Sterling pipeline/condition tree configurations. Map each to a business rule.
  • Agent inventory: Identify all time-triggered agents (e.g., payment collection, release agent, purge agent) — each becomes a scheduled microservice job.
  • DB2 schema reverse-engineering: Document all EXTN_FIELD customizations with business semantics.
  • Bounded context analysis: Use Domain Storytelling or Event Storming workshop with business to draw context boundaries.
  • Risk register: Identify top 10 failure scenarios from Sterling ops history (check incident log).
  • Traffic profiling: Peak RPS per API, peak order volumes by channel, seasonal patterns.

Phase 1 — Strangler Proxy + Infrastructure (Months 1–3)

  • Deploy Kong API Gateway in front of Sterling — all traffic proxied, zero logic change
  • Stand up Kafka (MSK), PostgreSQL (Aurora), Redis Cluster
  • Deploy Debezium CDC on Sterling DB2 → Kafka (read-only, no writes to new services yet)
  • Deploy Order Query Service reading from CDC events → MongoDB (shadow read model for validation)
  • Stand up EKS cluster, set up CI/CD pipelines, observability stack
  • Feature flag infrastructure (LaunchDarkly or AWS AppConfig)
  • Rollback: Remove Kong proxy → 100% traffic to Sterling. Recovery: <5 min
✅ Gate G1
Zero traffic disruption, P99 latency unchanged, CDC events flowing, shadow order read model <5% divergence from Sterling DB2.

Phase 2 — Extract Order Capture (Months 4–6)

  • Deploy Order Capture Service, Inventory Service, Payment Service
  • Route 5% of WEB channel new orders to OCS (feature flag: channel=WEB, pct=5)
  • OCS publishes order.created to Kafka; Sterling subscribes and processes these orders too (dual-path)
  • Compare outcomes: Sterling processing result vs new service for same order
  • Ramp from 5% → 25% → 100% over 6 weeks as divergence approaches 0
  • POS and marketplace channels remain on Sterling until Phase 3

Phase 3 — Order Orchestration + Fulfillment (Months 7–12)

  • Deploy Order Orchestrator (Temporal), Fulfillment Sourcing Service, Store Fulfillment Service
  • Shadow mode: run new sourcing engine in parallel with Sterling, log winner divergence
  • Onboard WEB + mobile channels to full new stack
  • Onboard POS after Store Fulfillment Service passes UAT
  • Data dual-write starts for order records (PostgreSQL + DB2)
  • Sterling transitions to read-only backup role for active orders

Phase 4 — Returns, Notifications, Shipping (Months 12–18)

  • Deploy Returns Management Service, Shipping Service, Notification Service
  • Onboard all remaining channels (marketplace, B2B)
  • Sterling DB2 write cutover → new PostgreSQL primary
  • Begin IBM WebSphere decommission process (license savings materialize here)

Top 3 Failure Runbooks

Runbook 1: Order Saga Stuck in PAYMENT_AUTHORISED

Symptom: Kafka consumer lag on order.events rising; orders not progressing to SOURCED state.
Action: Check Fulfillment Sourcing Service health → check Inventory Service Redis connection → check Drools rules load. If sourcing down >2min, activate Sterling fallback via feature flag. Compensate any stale sagas via admin API POST /v1/admin/sagas/{sagaId}/retry.

Runbook 2: Inventory Redis Cache Stampede

Symptom: Redis CPU >90%, Inventory Service P99 >500ms, orders failing soft-hold.
Action: Enable inventory.cache.bypass=false feature flag to route all reads directly to PostgreSQL (short-term). Scale Redis cluster horizontally via ElastiCache console. Check for missing TTL on keys (common cause: deployment without eviction policy).

Runbook 3: Kafka Consumer Lag > 50,000 (order.events)

Symptom: Orders are captured but not being processed; Saga orchestrator not receiving events.
Action: Check Temporal worker pod count → check Order Orchestrator deployment status → check MSK broker health. Scale up Order Orchestrator pods via kubectl scale deployment order-orchestrator --replicas=20. If >5min lag, activate Sterling processing fallback for new orders.

  • Reserved Instances (1-yr) for Order Capture, Inventory, PostgreSQL — always-on workloads

Security & Compliance

Authentication & Authorization

  • External channels: OAuth2 / OIDC via Cognito or Auth0. JWT tokens with short TTL (15 min).
  • Internal service-to-service: mTLS via Linkerd service mesh. Each service has a SPIFFE/SPIRE identity.
  • RBAC: Order read vs order mutate vs admin roles. Store associates have node-scoped access only.
  • Kafka: ACL per topic, service account per consumer group. No wildcard consumers.

PCI-DSS Compliance

⚠ PCI Scope Reduction
  • Payment Card data must NEVER flow through Order Capture Service. Use tokenization only — the payment token reference is passed, not card data.
  • Payment Service sits in a separate PCI-scoped VPC subnet with WAF + dedicated egress.
  • All services that touch payment tokens are PCI-scoped for penetration testing.
  • Sterling DB2 may contain historic PAN data — run a data discovery tool (e.g., Macie) before CDC replication starts.

India DPDP Act (Data Protection)

  • PII tokenization: Customer name, phone, email stored in a Customer Data Service. Order events carry only customerId (UUID). No PII in Kafka topics.
  • Data residency: All order and customer data in AWS ap-south-1 (Mumbai). No cross-region replication of PII without explicit consent.
  • Right to deletion: Deletion pipeline: DELETE /v1/customers/{id}/data triggers a saga that anonymizes all order records (name → "DELETED", address → lat/long centroid).
  • Consent logging: Every marketing opt-in/out stored in immutable OpenSearch index with timestamp.

Audit Logging

  • Every order state transition logged to immutable OpenSearch index: who, what, when, from_state, to_state, correlation_id
  • Audit logs shipped to S3 Glacier for 7-year retention (regulatory minimum)
  • WAF logs from Kong Gateway → OpenSearch for security analysis

Network Security

  • All services in private subnets. Only API Gateway has public-facing NLB.
  • VPC endpoints for S3, ECR, MSK (no public internet egress for data services)
  • Security Groups follow least-privilege: Order Capture only allows ingress from Kong, egress to PostgreSQL and Kafka.
  • AWS WAF on Kong: rate limiting per IP, SQL injection rules, geo-blocking for known fraud regions.

Architecture Decision Records

ADR-001 — Strangler Fig over Big-Bang Rewrite for Sterling Decomposition Accepted +

Sterling OMS handles 100% of order processing. A full rewrite would require feature-parity with years of accumulated business rules embedded in Pipeline configurations, condition trees, and DB-stored agent configs. Previous attempts at big-bang OMS rewrites in the industry (Macy's, Gap) ran 2–3 years over schedule.

Use Strangler Fig pattern with an API Gateway proxy layer in front of Sterling. Extract one bounded context at a time, route traffic incrementally via feature flags, maintain Sterling as a fallback path until new services achieve 99.9%+ reliability.

Pros: Zero-downtime migration; each extraction is independently rollback-able; value delivered incrementally from Month 6. Cons: Dual maintenance cost for 18 months; Kafka dual-path complexity; team must understand both Sterling and new stack simultaneously.

Big-bang rewrite: Rejected — unacceptable business risk, no rollback, feature-parity discovery takes 12 months alone. Lift-and-shift to cloud: Rejected — doesn't eliminate WebSphere/DB2 licensing or scalability constraints. Just moves the problem to EC2.

ADR-002 — Choreography + Orchestration Hybrid Saga (Temporal) over Pure Choreography Accepted +

Order lifecycle spans 8–12 services and can run for days (backorders, delayed shipments). Pure choreography (Kafka events only) produces spaghetti event dependencies and makes compensation debugging extremely difficult. Sterling's Pipeline model is effectively an orchestrator — business users expect clear state visibility.

Use Temporal.io as the saga orchestrator for the Order lifecycle. Individual domain events (payment.authorised, fulfillment.sourced) are still Kafka-based, but the Order Orchestrator service is a Temporal workflow that drives the sequence and manages compensations with explicit code, not implicit event chains.

Pros: Temporal provides built-in workflow history, retry logic, and a UI for ops visibility (near-equivalent to Sterling's pipeline monitor). Compensation is explicit code, not inferred. Cons: Adds Temporal cluster as new infrastructure dependency. Team needs to learn Temporal SDK (Go or Java).

Pure Kafka choreography: Rejected — compensation logic becomes implicit and untraceable. Debugging a stuck order requires reading 8 different topics. AWS Step Functions: Rejected — vendor lock-in, limited to AWS, expensive at scale (>1M state transitions/month), and lacks the Temporal developer ergonomics for complex long-running workflows.

ADR-003 — CQRS for Order Domain (Separate Write DB and Read Model) Accepted +

Order reads (customer order history, order status) have very different query patterns from order writes (create, state transitions). Customer-facing order history requires full-text search, date filtering, and status grouping — all terrible on a normalized transactional PostgreSQL schema. Mixing them causes lock contention during peak sale events.

Write path: PostgreSQL (normalized, ACID, saga state). Read path: MongoDB Atlas with a denormalized order document, populated via Kafka consumers. Order Query Service serves all customer-facing reads from MongoDB. Eventual consistency accepted (typically <2s lag).

Pros: Read path scales independently; MongoDB supports flexible querying for order history; write path is clean and ACID. Cons: Eventual consistency — customer may see stale status for 1–2s after state change. Read model is a derived projection — must be rebuildable from Kafka log if corrupt.

Single PostgreSQL for reads and writes: Rejected — read replicas do not solve the query flexibility problem; complex analytics queries would require full table scans on order lines. Elasticsearch for read model: Considered but rejected for primary read model — higher operational complexity than MongoDB for document-style access patterns.

ADR-004 — Outbox Pattern for Order Capture → Kafka Reliability Accepted +

If Order Capture Service writes to PostgreSQL and then publishes to Kafka, there is a window where the DB write succeeds but the Kafka publish fails — producing a ghost order that never gets processed. This is an unacceptable data loss scenario (RPO = 0 for order records).

Implement Transactional Outbox Pattern: Order Capture writes the order row AND an outbox event row in a single PostgreSQL transaction. A separate Outbox Relay service (or Debezium) reads the outbox table and publishes to Kafka. The event is only published after it's durably written to the DB.

Pros: Zero risk of order loss; Kafka publish is eventually consistent but guaranteed. Replay is possible by re-processing outbox rows. Cons: Adds ~5ms latency to order capture. Outbox table grows and needs periodic cleanup (archive processed rows to S3).

Direct Kafka publish then DB write: Rejected — inverse problem: published event with no DB record produces phantom orders. Two-phase commit (XA transactions): Rejected — PostgreSQL and Kafka do not support XA; WebSphere XA is exactly the complexity we're escaping from Sterling.

ADR-005 — Drools Rules Engine for Fulfillment Sourcing (not hardcoded logic) Accepted +

Sterling's Sourcing Rules Engine is configured by operations and supply chain teams via a UI — non-developers change sourcing priorities, node exclusions, SLA rules daily. This business agility must be preserved in the new stack. If we hardcode sourcing logic in Java, we reintroduce the Sterling deployment bottleneck for business rule changes.

Use Drools (OSS) or OpenL Tablets as the rules engine within the Fulfillment Sourcing Service. Business rules (node priorities, SLA tables, exclusion rules) are stored in a rules repository and hot-reloaded without service restart. Sourcing engineers can modify rules via a thin admin UI without engineer involvement.

Pros: Business agility preserved; non-developer rule changes without deployment; auditable rule change history. Cons: Rules engine is an additional dependency; complex rules are harder to unit test; Drools has a steep learning curve.

Hardcoded Java logic: Rejected — defeats the purpose; every sourcing rule change requires a developer and a deployment. ML-based sourcing: Considered for future phase — good for optimization, but not a replacement for deterministic rule-based sourcing that supply chain teams can audit and trust.

⚖️ Trade-offs to Verbalize Proactively

  • Eventual consistency in CQRS: Customer-facing order status may lag 1–2s. Acceptable for most retail, not for real-time POS.
  • Temporal vs pure choreography: Orchestration gives visibility but adds infra; choreography is simpler but harder to debug at scale.
  • 18-month dual maintenance: Both Sterling and new stack run in parallel. This has a real cost — budget for it explicitly.
  • Drools rules engine: Gives business agility but adds operational complexity. Right for sourcing; wrong for simple discount logic.
  • DPDP tokenization: Adding a Customer Data Service adds a network hop to every order capture. Worth the PII compliance benefit.
