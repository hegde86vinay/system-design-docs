Pattern Overview — What is the Anti-Corruption Layer?

The Anti-Corruption Layer (ACL) is a Domain-Driven Design (DDD) pattern introduced by Eric Evans in "Domain-Driven Design: Tackling Complexity in the Heart of Software." It is a translation and isolation boundary placed between two systems with fundamentally different domain models, vocabulary, and semantics — preventing one system's concepts from "corrupting" the design of the other.

Core Definition
The ACL acts as a bidirectional translator — it speaks the language of the legacy system on one side, and the language of your new clean domain model on the other. It is not a proxy, not a facade, and not just an API gateway. It is a semantic translation boundary.

What ACL IS

  • A semantic translator between domain models
  • An isolation shield for your new bounded context
  • A protocol/format converter (XML ↔ JSON, SOAP ↔ REST)
  • A data model transformer (Sterling XML schema → canonical domain objects)
  • A contract adapter that decouples your release cycle from legacy
  • A fault-isolation boundary (legacy failures don't cascade)

What ACL is NOT

  • Not an API Gateway (no routing/auth concern)
  • Not a Facade (Facade wraps, ACL translates)
  • Not just a data mapper / DTO converter
  • Not a permanent architecture component (it should be eliminated as legacy is decommissioned)
  • Not a "fix" for bad legacy design — it is protection from it
  • Not a place for business logic

The DDD Context Map Position

In DDD Context Map terms, ACL represents a relationship between a Downstream (your new microservice) and an Upstream (Sterling OMS legacy) where the downstream team cannot influence the upstream model. The ACL is the downstream's protection mechanism.

The Three Core Responsibilities

TRANSLATE
Convert Sterling's domain model (YantraCommerceOrder, DocumentType, EnterpriseCode) into your canonical domain model (Order, OrderStatus, Tenant). Map field-by-field with semantic alignment, not just renaming.
ISOLATE
Ensure your new microservices never import Sterling's Java classes, XML schemas, or domain vocabulary directly. All coupling is confined inside the ACL module. New services speak only your domain language.
STABILIZE
Present a stable interface to downstream consumers even when Sterling changes (version upgrades, schema changes, agent behavior changes). The ACL absorbs upstream volatility so new services don't need to change.

Why ACL is Critical for IBM Sterling OMS Modernization

IBM Sterling OMS carries 20+ years of enterprise retail domain model evolution baked into its XML schema, Java APIs, and database design. Integrating directly with Sterling from new microservices without an ACL is one of the most common and costly architectural mistakes in retail modernization programs.

Sterling's "Corrupting" Characteristics

Sterling Artifact Problem for New Microservices ACL Mitigation
YantraCommerceOrder XML Monolithic 500+ field XML document — you pull the whole payload for one field ACL extracts only needed fields into a lean canonical Order object
DocumentType=0001 / 0003 Magic strings as type identifiers — meaningless in your domain ACL maps to typed enums: OrderType.SALES_ORDER, PURCHASE_ORDER
EnterpriseCode + SellerOrganizationCode Dual-key organization model that conflicts with modern tenant/org models ACL normalizes to tenantId per your data model
SOAP/HTTP Agent APIs + invokeService() RPC-style coupled calls; no events; tight latency coupling ACL wraps in async adapters; emits domain events over Kafka
OrderLineKey as surrogate PK Internal Sterling UUID — meaningless externally, not portable ACL generates your own lineId; maintains Sterling key in private mapping table
ExtnOrderLine.* extension columns Schema extensions with unbounded attribute names — no type safety ACL maps to first-class typed fields in canonical schema
Order Status as 1100.100 style codes Numeric status hierarchy hard to parse; status semantics unclear ACL maps to OrderStatus enum: CREATED, RELEASED, SHIPPED etc.
JMS/MQ-based agent event notifications Message format tied to Sterling's internal object model ACL subscribes to MQ, transforms to clean domain events on Kafka
Agent-based processing (Task Q / Time-Triggered) Polling-based processing difficult to observe or replace incrementally ACL exposes agent trigger hooks as event sources your new services can consume
Real-World Anti-Pattern Without ACL
Teams that skip the ACL and integrate directly with Sterling end up with Sterling XML schema types leaking into new service code, leading to: new services that can't be tested without a Sterling instance, domain models that look like Sterling's YantraCommerceOrder with a React frontend on top, and migration that becomes impossible because every new service has Sterling coupling embedded.

The Strangler Fig + ACL Combination

ACL works as the translation backbone of the Strangler Fig migration pattern. As each bounded context is carved out of Sterling into a new microservice, the ACL is the bridge that allows both systems to coexist during the transition window. The ACL is the "fig" that lets the new vine grow while the old tree still stands.

ACL as a Dedicated Microservice

Deployment Decision
The ACL should be deployed as a dedicated, independently deployable microservice — not as a library embedded in each new service, and not bundled inside Sterling itself. This isolates the translation concern, allows independent versioning, and makes the ACL easy to remove once Sterling is decommissioned.

ACL as a Service — Pros

  • Single place for all Sterling translation logic
  • Independent deployable — can be updated without touching new services
  • Can be horizontally scaled if Sterling call volume is high
  • Clear decommission target when Sterling goes away
  • Can own the key-mapping database independently
  • Allows different teams to own ACL vs new domain services

ACL as a Service — Cons / Mitigations

  • Additional network hop → mitigate with async/event-driven patterns, not synchronous chains
  • Single point of failure → mitigate with HA deployment, circuit breakers, fallback to cache
  • Team ownership confusion → assign explicit ownership; treat it as part of platform team's portfolio
  • Performance bottleneck → async Kafka-based translation for non-latency-critical flows

ACL Internal Structure — Components in Detail

ADAPTER Protocol / Transport Adapter
Handles the protocol mismatch. Knows how to talk to Sterling — SOAP calls via invokeService(), JMS/MQ message consumption, DB2 CDC events via Debezium. Converts raw Sterling wire format (XML, MQ bytes) into an internal ACL representation. Does NOT translate domain semantics — that's the Translator's job.
TRANSLATOR Domain Model Translator
The heart of the ACL. Converts Sterling's domain model artifacts (YantraCommerceOrder, DocumentType codes, status strings) into your canonical domain model objects (Order, OrderStatus enum, etc). Contains all field mapping logic. Should be stateless, pure functions: Order translate(SterlingOrderDTO sterlingOrder).
KEY MAPPER Bidirectional Key Store
Maintains a persistent mapping between Sterling's internal keys (OrderHeaderKey, OrderLineKey) and your canonical UUIDs (orderId, lineId). Critical for write-back flows: when your new service updates an order, the ACL must resolve back to Sterling's key. Backed by Redis (fast lookups) + PostgreSQL (durability).
FACADE Stability Shim / Versioned Contract
Exposes a stable, versioned contract to downstream new services — REST or gRPC. Acts as a buffer: when Sterling upgrades its schema or Sterling 9.3 → 10.x migration happens, only the ACL's internal adapter changes. New services continue calling the same stable ACL API. Equivalent to an "Open Host Service" contract for your new bounded context.

Scenario 4: Post-Purchase — Order Cancellation

Customer initiates cancellation from self-service portal. The new Order Service validates eligibility (pre-ship only), then issues a cancel command to the ACL. The ACL translates to Sterling's changeOrder with Action=CANCEL, handles the synchronous response, and emits a canonical event.

Sterling Cancellation Eligibility Window
Sterling enforces cancellation only before the order is in Pick Confirmed state (3200.100). The ACL must check the cached order status before calling Sterling — if status is already PICKED or beyond, it returns a 409 Conflict to the new service without calling Sterling at all.

// ACL Cancel Command Handler
public CancelOrderResult cancelOrder(CancelOrderCommand cmd) {
    // 1. Idempotency check
    Optional<CancelOrderResult> cached = idempotencyStore.get(cmd.getIdempotencyKey());
    if (cached.isPresent()) return cached.get();

    // 2. Resolve Sterling key
    String sterlingKey = keyMapper.toSterlingKey(cmd.getOrderId());

    // 3. Build and invoke Sterling changeOrder
    String xml = sterlingXmlBuilder.buildCancelXml(sterlingKey, cmd.getReason());
    SterlingResponse resp = soapAdapter.invokeService("changeOrder", xml);

    // 4. Translate response
    CancelOrderResult result = translator.translateCancelResponse(resp, cmd.getOrderId());

    // 5. Store idempotency + publish event (in same outbox transaction)
    outboxService.publishAndStore(cmd.getIdempotencyKey(), result,
        new OrderCancelledEvent(cmd.getOrderId(), cmd.getReason()));

    return result;
}

The Corruption Cascade Without ACL

When the Order Service integrates directly with Sterling, the corruption cascades outward. The Notification Service needs to display order status — so it reads from the Order Service DB, which now stores 9000.100 status strings. The Analytics Service queries order events, which contain OrderHeaderKey alongside orderId. The Returns Service needs to cancel — so it also imports Sterling SOAP. Within 12 months, every service in the platform speaks Sterling. Decommissioning Sterling becomes equivalent to rebuilding the entire platform.

Scenario 4 — Full Migration Exit: Moving Away from Sterling Completely

This is the endgame of the modernization program. The ACL made it possible to build new services without Sterling coupling during the migration. Now the goal is to eliminate Sterling as the system of record for cancellations, move the cancel logic fully into the new Order Service, and retire the ACL's cancel adapter. Stage-by-Stage Exit Playbook — Cancellation Scenario

STAGE 1  Current State — Sterling is System of Record
Who owns cancel: Sterling OMS via ACL.
Order Service role: Sends cancel commands to ACL; receives translated Kafka events back.
Exit criteria to Stage 2: New Order Service has its own orders table with full order state; cancel eligibility logic implemented natively; all cancellations dual-write tested in staging.
STAGE 2  Dual Write — New DB and Sterling Written Simultaneously
Who owns cancel: Both — ACL writes to Sterling AND the new Order DB in the same request.
ACL role: After calling Sterling successfully, ACL also writes a canonical cancel record to the new Order DB. Two writes, one ACL request.
Reconciliation job: Nightly batch compares Sterling's cancelled order list against the new Order DB. Any divergence page-alerts the platform team.
Exit criteria to Stage 3: Reconciliation runs for 2+ weeks with divergence rate below 0.01%. New Order DB is confirmed the richer, more accurate source.
STAGE 3  New Service Leads — Sterling Becomes Shadow Write
Who owns cancel: New Order Service. Sterling write is async fire-and-forget via ACL.
Order Service role: Writes cancel directly to its own DB; publishes order.cancelled Kafka event immediately. Also sends an async cancel command to ACL which propagates to Sterling in the background (no customer-facing latency impact).
Feature flag: Toggle that cuts the Sterling shadow write off without a deployment. Used to measure Sterling write failure rate with zero customer impact.
Exit criteria to Stage 4: Sterling shadow write failure rate is negligible (Sterling can no longer block or slow the cancel flow). All downstream systems confirmed reading from Kafka / new Order DB — not from Sterling MQ.
STAGE 4  Full Exit — Sterling Retired, ACL Cancel Adapter Shut Down
Who owns cancel: New Order Service entirely. Sterling does not exist in this flow.
Steps to retire:
  1. Turn off feature flag for Sterling shadow write — observe zero impact for 72 hours.
  2. Shut down ACL's SterlingCancelAdapter — remove from Kubernetes deployment.
  3. Archive key-mapping table rows for cancelled orders (audit/compliance retention).
  4. Confirm zero consumers reading from Sterling's OMS.ORDER.CANCEL MQ queue.
  5. Decommission Sterling's YFS_ORDER_CANCEL related DB tables from read path.
  6. Remove ACL cancel endpoint from OpenAPI spec; notify downstream teams.

The ACL was never meant to be permanent. It was the scaffolding that let the new Order Service grow without touching Sterling's internals. By the time Stage 4 is reached, the new Order Service has a clean cancellation model, idempotency, event publishing, and eligibility logic — all built and battle-tested during the 18+ months the ACL was live. The ACL is removed not because it failed, but because it succeeded: it protected the new domain model long enough for it to stand on its own. The ACL is the chrysalis that makes Sterling decommission possible.

Scenario 5: Post-Purchase — Change Fulfillment Node (Ship-From-Store Re-route)

Customer or CS agent requests a fulfillment node change — e.g., re-route from a DC to a closer store. Saga Compensation on Partial Failure
If Step 1 (backorder) succeeds but Step 2 (re-release) fails, the ACL must execute a compensating transaction — re-release to the original ShipNode. The ACL owns this saga coordination. The Fulfillment Service sees it as an atomic operation: either fully re-routed or not at all. Never expose Sterling's two-step internals to the new service.

GraphQL ACL Pattern — Beyond REST

The ACL pattern is protocol-agnostic. While REST is natural for synchronous command APIs, GraphQL is increasingly the right choice for the ACL's query surface — especially when multiple new microservices need to compose Sterling-derived data differently (mobile app, CS portal, analytics dashboard). GraphQL lets each consumer fetch exactly what it needs without the ACL maintaining dozens of REST endpoint variants.

When to Use GraphQL vs REST in the ACL
Use GraphQL for the ACL's query/read surface — rich order detail composition, multi-entity lookups, consumer-specific projections. Use REST for the ACL's command/write surface — cancel, reroute, change-address, change-LOS. Commands have deterministic payloads; queries benefit from GraphQL's field-selection flexibility.

GraphQL Resolver with Sterling Translation


// Spring GraphQL Resolver in ACL service
@QueryMapping
public Order order(@Argument String orderId) {
    // 1. Resolve Sterling key
    String sterlingKey = keyMapper.toSterlingKey(UUID.fromString(orderId));

    // 2. Fetch from Sterling SOAP (or read-cache if available)
    SterlingOrderDTO sterlingOrder = soapAdapter
        .invokeService("getOrderDetails", sterlingKey);

    // 3. Translate — Sterling XML → canonical Order domain object
    return orderTranslator.translate(sterlingOrder);
    // Returns Order with clean enum statuses, UUID keys, Money types
    // Zero Sterling field names visible to the GraphQL consumer
}

// DataLoader for batch line fetching (prevents N+1 on OrderLine queries)
@BatchMapping
public Map<Order, List<OrderLine>> lines(List<Order> orders) {
    List<String> sterlingKeys = orders.stream()
        .map(o -> keyMapper.toSterlingKey(o.getOrderId()))
        .collect(toList());

    // Batch SOAP call to Sterling getOrderLineList
    Map<String, List<SterlingOrderLine>> sterlingLines =
        soapAdapter.batchGetOrderLines(sterlingKeys);

    return orders.stream().collect(toMap(
        o -> o,
        o -> lineTranslator.translate(
            sterlingLines.get(keyMapper.toSterlingKey(o.getOrderId())))
    ));
}

GraphQL ACL — Key Design Rules

Do

  • Use DataLoader for all Sterling SOAP calls to prevent N+1 — batch by OrderHeaderKey[]
  • Cache translated responses in Redis with TTL keyed to orderId — Sterling reads are expensive
  • Version the GraphQL schema (@deprecated directives) independently of Sterling upgrades
  • Define all types in canonical domain vocabulary — no Sterling field names in the SDL
  • Use persisted queries for production to enable CDN caching and prevent query injection

Do NOT

  • Use GraphQL mutations for post-purchase write commands — REST gives clearer idempotency and error semantics
  • Expose Sterling's raw XML as a GraphQL scalar type (even as a fallback) — this breaks ACL isolation
  • Allow unbounded depth queries against Sterling — set maxDepth and maxComplexity limits
  • Skip DataLoader — without it, a 10-line order query triggers 10 individual Sterling SOAP calls
  • Use GraphQL subscriptions directly over Sterling MQ — use Kafka as the event backbone; subscribe to Kafka, not MQ
ADR-005 — GraphQL for ACL Read Surface, REST for Write Commands Accepted +

The ACL serves multiple downstream consumers (mobile app, CS portal, BFF, analytics). Each needs different field projections from Sterling order data. Using REST, the team was proliferating narrow endpoints. Mutations (cancel, reroute) need clear HTTP status semantics and idempotency key support.

Expose a GraphQL endpoint (/acl/graphql) for all read/query operations over Sterling-derived data. Continue using REST endpoints (/acl/commands/*) for all write-back commands (cancel, reroute, change-LOS, change-address). The ACL owns one schema for each.

Pros: Consumers self-select fields — mobile stays lightweight; CS portal gets full detail. DataLoader prevents N+1 Sterling SOAP calls. Single SDL schema evolves independently of Sterling. Cons: GraphQL runtime adds operational surface; CDN caching requires persisted queries; team must learn DataLoader patterns.

GraphQL mutations for commands: Rejected — Sterling write-back failures need HTTP 409/422 status codes, Retry-After headers, and idempotency-key request headers. GraphQL mutations don't support these HTTP-level semantics cleanly. REST-only: Rejected — would require maintaining 12+ narrow query endpoints across three consumer types, all doing similar Sterling translation with different field subsets.

Non-Negotiable ACL Design Rules

  1. No Business Logic in ACL

    The ACL translates — it does not decide. If you find yourself writing if/else business rules inside the ACL, that logic belongs in the new domain service. ACL is a dumb pipe with semantic awareness.

  2. New Services Must Never Import Sterling Classes

    Your Order Service must have zero compile-time dependency on Sterling's JAR files or XML schemas. If a Sterling class appears in any new service's import, the ACL boundary is broken.

  3. ACL Must Be Testable Without Sterling

    All translation logic should be unit-tested with stubbed Sterling responses. The SOAP/MQ adapter can be mocked. This is your canary — if you can't test translation without Sterling, you have a coupling problem.

  4. Idempotency on Both Read and Write Paths

    Sterling calls can fail and be retried. Every write-back to Sterling via the ACL must carry an idempotency key. Every event translation must be deduplicated (event ID store in Redis with TTL).

  5. Observability: Trace Across the ACL Boundary

    Propagate traceId across the ACL. When a Sterling event enters the ACL and a Kafka event exits, the same traceId must be visible in both. This is essential for debugging post-purchase modification failures that span Sterling + new services.

ACL Lifecycle — The Most Important Rule: Plan Its Removal

Architectural Anti-Pattern: Permanent ACL
The most dangerous ACL failure mode in enterprise modernization is when the ACL becomes permanent infrastructure because legacy was never decommissioned. You end up maintaining three systems: Sterling, ACL, and new services — all indefinitely. The ACL must have a defined sunset date from Day 1.

ACL Removal Checklist

  1. Every Sterling-originated event has a canonical replacement

    New services must own and emit their own domain events. When the last event that was originally sourced from Sterling is now sourced from a new service, that ACL adapter can be turned off.

  2. Zero remaining Sterling write-back commands

    All post-purchase modifications, cancellations, and fulfillment triggers must originate from new services directly — not through ACL → Sterling. Sterling must be fully decommissioned as the system of record before ACL can be removed.

  3. Key mapping store migrated

    All historical Sterling key → canonical UUID mappings must be migrated to a permanent read-only archive in the new data platform for audit/compliance. The ACL's Redis/Postgres key store should not be deleted — archived.

  4. Consumer lag = 0 on all ACL-published Kafka topics

    Ensure no consumers are still depending on ACL-published events before shutting down ACL Kafka publishers. Use consumer group monitoring dashboards.

  5. Traffic reconciliation report signed off

    Compare order volume and status transitions in new services vs. what Sterling processed in the equivalent period. No unexplained discrepancies before cutting over.

ACL Debt Metrics — Track These Weekly

Translation Coverage

  • % of Sterling event types with ACL translators
  • % of Sterling API operations wrapped in ACL adapters
  • Translation error rate (failed parses / total events)
  • Untranslated field count per event type

Migration Progress

  • % of order volume flowing through new services vs Sterling
  • Active ACL adapters still running (reduce to 0)
  • Sterling API call volume (reduce to 0)
  • ACL write-back command volume (reduce to 0)

Architecture Decision Records

ADR-001 — ACL as Dedicated Microservice vs. Library per New Service Accepted +

New microservices need to interact with Sterling during the migration period. The ACL translation logic must live somewhere — either as a shared library in each service or as a centralized service.

Deploy ACL as a dedicated, independently deployable microservice with its own Kubernetes deployment, database, and Kafka topics. New services never import Sterling libraries.

Pros: Single decommission target; independent deployable; clear ownership; no Sterling class leakage into new services. Cons: Additional network hop for synchronous calls; another service to operate.

Library embedded per service: Rejected — Sterling schema changes would require updating and redeploying every service. Sterling classes leak into domain code. ACL inside Sterling itself: Rejected — requires Sterling customization deployment for every change; violates isolation principle.

ADR-002 — Async Event-Driven ACL vs. Synchronous ACL Proxy Accepted +

The ACL can operate synchronously (new service calls ACL, ACL calls Sterling, returns response) or asynchronously (ACL translates Sterling events and publishes to Kafka, new services consume from Kafka).

Default to async event-driven for all read/event flows (MQ → ACL → Kafka → new services). Use synchronous ACL only for write-back commands where the caller needs an immediate acknowledgment (e.g., order cancellation confirmation).

Pros: New services are decoupled from Sterling's availability; Kafka provides event replay for bootstrap; async is naturally more resilient. Cons: Eventual consistency for read operations; need to handle out-of-order events.

Fully synchronous proxy: Creates a latency chain Sterling → ACL → new service. Sterling downtime immediately impacts new services. Eliminates resilience benefits of microservices architecture.

ADR-003 — Key Mapping Store: Redis + PostgreSQL vs. Single Store Accepted +

ACL needs to maintain bidirectional key mappings between Sterling's internal keys and canonical UUIDs. This store must be fast (order of milliseconds for lookup on hot paths) and durable (cannot lose mappings).

Dual-store: Redis for sub-millisecond lookups on hot translation paths; PostgreSQL as durable source of truth with full audit history. Redis is populated from Postgres on startup and kept warm during operation.

Pros: Fast lookups without Postgres round-trip on every event; durable audit trail; Redis can be rebuilt from Postgres if evicted. Cons: Two stores to operate; need cache invalidation logic.

Redis only: Risk of key loss on Redis failure/eviction with no recovery path. Postgres only: Acceptable for low-volume but adds 5-10ms per translation — unacceptable for high-throughput event flows.

ADR-004 — Transactional Outbox in ACL for MQ → Kafka Reliability Accepted +

When the ACL receives a Sterling MQ event and needs to publish a canonical domain event to Kafka, there is a dual-write problem: MQ acknowledgment and Kafka publish can diverge if the ACL crashes mid-operation.

Implement the Transactional Outbox pattern: write the translated canonical event to a PostgreSQL outbox table in the same transaction as key mapping updates. A separate outbox relay process (Debezium CDC) reads the outbox and publishes to Kafka. MQ message is acknowledged only after outbox write succeeds.

Pros: At-least-once delivery guarantee; no lost events even on ACL crash; exactly-once effectively achieved with downstream idempotency. Cons: Additional Postgres table; Debezium relay process to operate.

Dual-write without outbox: ACL publishes to Kafka then ACKs MQ. If Kafka publish succeeds but ACL crashes before MQ ACK, Sterling re-delivers causing duplicate events. Without outbox, reliability is non-deterministic.

Interview Cheat Sheet — Principal / Director Level

🎯 Top Clarifying Questions

  • Is Sterling the system of record for orders during migration, or are we running dual-write?
  • What is the target decommission date for Sterling? (ACL is temporary — what's the sunset?)
  • Are there real-time SLA requirements on the Sterling integration, or can we accept eventual consistency?
  • Which Sterling APIs are in-scope: are we integrating with 9.3, 10.x, or both?
  • What existing IBM MQ topology exists — are agents publishing to known queues already?

⚖️ Key Trade-offs to Verbalize

  • ACL adds latency → mitigate with async-first design; sync only for write-backs
  • ACL can become permanent if Sterling isn't decommissioned → need hard migration milestones
  • Async ACL creates eventual consistency → explicit consistency contracts with downstream consumers
  • Key mapper is a single point of failure → HA Redis + Postgres backup, pre-warm on startup
  • Translation errors silently drop events → DLQ + alerts on every translator

🔍 Failure Modes Interviewers Probe

  • "What happens if Sterling is down — can your new services still serve orders?"
  • "What happens if the ACL crashes mid-translation — do you lose events?"
  • "What if a new order ID in your system has no matching Sterling key yet?"
  • "How do you handle Sterling schema changes in a future Sterling upgrade?"
  • "Your ACL has been running for 2 years — how do you prevent it becoming permanent?"

✨ "Wow" Depth Signals

  • Distinguish ACL from Facade: ACL translates domain semantics, Facade only wraps API surface
  • Call out Sterling's status hierarchy (1100.100 etc.) and the semantic mapping challenge — most candidates don't know this
  • Propose Transactional Outbox pattern for MQ → Kafka reliability — shows event architecture maturity
  • Name the ACL's own sunset as a first-class architectural concern; show a decommission checklist
  • Mention that ACL should emit Kafka events with the same traceId as the originating Sterling request — for end-to-end distributed tracing across the legacy boundary

🚫 Common Mistakes to Avoid

  • Putting business logic in the ACL (validation, rules, calculations)
  • Treating ACL as an API gateway (they solve different problems)
  • Not planning for the ACL's own removal — making it permanent infrastructure
  • Synchronous ACL chain for all calls — creates a latency bottleneck and Sterling-as-SPOF
  • Letting Sterling classes leak into new service codebases via shared libraries

❓ Follow-up Questions to Ask Interviewer

  • "What's the organization's tolerance for eventual consistency on post-purchase flows — is real-time modification confirmation required by the customer-facing app?"
  • "Is there an existing Kafka / event backbone in the GCC, or would this be greenfield?"
  • "Has the team considered using Temporal.io as a replacement for Sterling's Task Queue agents — and would the ACL need to bridge Sterling agents to Temporal workflows?"
One-Line Definition to Open With in Any Interview
