GIV Architecture Overview

The Global Inventory Visibility (GIV) module is the inventory brain of Sterling OMS. It maintains a real-time picture of all supply and demand across every ship node — stores, warehouses, and vendor/drop-ship locations — and provides availability answers to the order promising, scheduling, and release engines.

Core Responsibility
GIV answers one fundamental question: "For this item, at this node (or across nodes), how much inventory can I promise to a new demand, given all existing supplies and committed demands?"

Three Levels of Inventory Availability

Node Level

ATP at a specific ship node (a single store or warehouse). Most granular. Used during scheduling and release.

getATP(shipNode=NODE1)
Distribution Group Level

Aggregate ATP across a defined group of nodes (e.g., all Northeast warehouses). Used for regional promising.

getATP(distributionGroup=DG1)
Enterprise Level

Total ATP across all nodes for an item under an enterprise. Broadest view, used for catalog-level availability displays.

getATP(enterpriseCode=ENT1)
Key Gotcha
Node-level sum ≠ Enterprise-level ATP. Unassigned demands (OPEN_ORDER with no shipNode stamped) are subtracted at enterprise level but NOT at node level — so summing node ATPs always overstates enterprise availability. Always use getATP for enterprise availability, not getSupplyDetails sum.

The ATP Formula

Available-To-Promise (ATP) is the quantity of an item available to commit to a new incoming demand, after accounting for all existing supplies and demands within a configured time horizon.

ATP = Total Qualifying Supply − Total Qualifying Demand
Evaluated across a configured date range, per the Inventory Considerations matrix in GIV

What "Qualifying" Means

Not every supply type and not every demand type enters the ATP formula. The GIV Inventory Considerations screen maps which supply types and which demand types are considered together, depending on the incoming demand type being evaluated.

Classic Example
When the input DemandType = OPEN_ORDER (a new web order), the system considers: Onhand + Firm Plan + PO + PO Placed + PO Released + PO Scheduled supplies, and ALL existing demands. When evaluating DemandType = ALLOCATED (for release), only Onhand + WIP supplies are considered.

ATP Calculation Steps

1
Determine Evaluation Horizon The date range is derived from lead time configuration + requested ship date. Supplies and demands outside this window are excluded.
2
Collect Qualifying Supplies Look up YFS_INVENTORY_SUPPLY records for the item-node(s) filtered by supply types configured in Inventory Considerations for the incoming DemandType.
3
Collect All Demands All existing demands from YFS_INVENTORY_DEMAND are summed — OPEN_ORDER, BACKORDER, RESERVED, SCHEDULED, ALLOCATED — including unassigned demands at enterprise level.
4
Apply Safety Factor A percentage of the supply is withheld based on the safety factor configuration per item/node. This reduces the effective supply available for promising.
5
Apply Expiration Logic Expired supply quantities (past item-level expiration time) are deducted from supply before the formula runs. Expired items are not counted.
6
Compute ATP = Supply − Demand Final ATP = qualifying supply (after safety factor and expiration) minus total qualifying demands. Result can be negative (backorder condition).

Don't confuse these. The Availability Type at the node level (TRACK / INFINITE) controls whether Sterling tracks inventory for that node at all. Warehouse Supply Management

Warehouses (Distribution Centers) are the primary ONHAND supply source. Key characteristics:

  • WMS Integration: WMS systems feed inventory adjustments via adjustInventory API or batch file. In 9.3, this is often an IBM MQ-based async feed.
  • PO Supply: When a Purchase Order is received in ERP (SAP/Oracle), Sterling creates PO_PLACED → PO_RELEASED → PO_SCHEDULED supply records, advancing through states as the PO progresses.
  • On Receipt: When the PO is received at the DC, the PO supply record is converted to ONHAND supply (net increase), and the PO supply record is decremented/deleted.
  • Availability Type: Always TRACK. Every unit matters and is tracked.

Store Supply Management

Stores as fulfillment nodes (Ship-from-Store, BOPIS) present unique challenges:

  • Intraday Feed: Stores typically push delta inventory feeds (e.g., every 30 min or hourly) via adjustInventory API. Sterling does not connect to store POS in real-time in 9.3 without custom integration.
  • Null Pick Problem: Store inventory can be inaccurate (shrinkage, theft, misplaced). When a store associate cannot find inventory ("null pick"), the Dirty Node mechanism is triggered via manageInventoryNodeControl — that node is excluded from sourcing for that item until the inventory picture is corrected.
  • Store Returns: When an item is returned to a store and returned-to-shelf, adjustInventory with positive quantity adds ONHAND supply back. Note: returns that go to a back-room hold may use a separate supply type or segment.
  • Short Pick / Backorder: If a store cannot fulfill a released order (short pick via Sterling Store module via changeShipment), the demand should transition to BACKORDER. A known 9.3 issue is that the shipNode on the order line is NOT cleared, causing the BACKORDER demand to remain node-locked, leading to incorrect RTAM ATP publication.

Vendor / Drop-Ship Supply Management

Vendor nodes have two modes in Sterling OMS:

  • INFINITE Node: The vendor node is configured as AvailabilityType = INFINITE. Sterling never backorers at this node — all incoming orders are scheduled against it. ATP is always "yes". Used when vendor manages their own availability and Sterling just routes.
  • TRACKED Vendor Node: Vendor sends inventory advice (EDI 846 or API feed), which is loaded as supply records. Sterling tracks PO and available quantities. More accurate but requires vendor cooperation.
  • Chained PO Flow: When a sales order is sourced to a vendor node, a chained Purchase Order is created (createChainedOrder). Sterling inserts a PROCUREMENT_PLACED synthetic supply at the ship node and a corresponding demand. When the chained PO is scheduled and received, these synthetic records are cleaned up.
  • Drop Ship Confirmation: Vendor confirms shipment via EDI 856 (ASN) or API, triggering the demand to be consumed and ONHAND decremented.

Distribution Groups & Node Aggregations

Distribution Groups (DGs) are the core aggregation mechanism in GIV. A DG is a logical grouping of ship nodes used for both sourcing rules and inventory availability aggregation.

Default Distribution Group

Every enterprise has a Default DG. When no specific DG is configured for an item or sourcing rule, the default DG is used for RTAM monitoring and availability publication.

Node-Level Distribution Group

A specific DG assigned to a subset of nodes. Used to create regional availability pools (e.g., all stores in Maharashtra, all East Coast DCs).

How DGs Affect ATP

When getATP is called with a Distribution Group, the engine:

  • Sums ONHAND + future supplies across all nodes in the DG
  • Sums all assigned demands across all nodes in the DG
  • Handles unassigned demands at the DG level separately (the backorder distribution problem)
  • Applies safety factors per item-node configuration before summing
Backorder Demand Distribution Problem (v9.3)
When a backorder demand has no shipNode (unassigned), RTAM must distribute it across nodes in a DG to calculate node-level ATP. In 9.3, this distribution is random — RTAM may assign the backorder demand to Store 5 for RTAM ATP calculation, even though the Schedule Agent later routes the actual order to Store 1. This causes transient ATP inaccuracies in RTAM publish. The priority-based unassigned demand distribution feature was added in 9.5 to address this.

Three Monitor Types in GIV

Onhand Inventory Monitor

Triggers when onhand quantity crosses a configured threshold (e.g., falls below 10 units). Used for replenishment triggers and low-stock alerts.

Event-Based Availability Monitor (RTAM)

Triggers on every inventory change event. Recalculates ATP at item-DG level and publishes to subscribed external systems. This is the standard channel availability sync mechanism.

Action-Based Availability Monitor

Triggers configurable actions (custom Java class, API call) when inventory crosses thresholds. Used for automated replenishment order creation (transfer orders).

RTAM ATP vs. Order-Time ATP

Critical Distinction
RTAM publishes a snapshot ATP — not a live lock. Between the time RTAM publishes ATP=5 and a customer places an order, other orders may arrive. RTAM-based ATP is used for display availability (website, app) while order-time ATP (via getATP/findInventory called synchronously during checkout) is the authoritative source. If the channel relies solely on the RTAM cache for order acceptance without a live check, overselling is possible during high-velocity periods.

RTAM Performance Considerations (9.3)

  • RTAM agent is an Integration Server process — configure thread counts based on expected inventory change throughput
  • For Hot SKUs (high-velocity items), Sterling 9.3 introduced the Hot SKU property set to minimize DB locking during concurrent inventory reads/writes
  • Use getAvailabilityCache (short-lived cache) for product browsing pages to reduce RTAM load
  • Smart Sourcing + GIV-RTAM with node-level monitoring: use RTAM node-level ATP to filter out empty nodes from sourcing, avoiding unnecessary Schedule Agent compute

Hot SKU & Performance Tuning

A Hot SKU is an item with extremely high concurrent order volume — flash sale items, free gift items, deeply discounted products. Without special handling, concurrent DB reads/writes to YFS_INVENTORY_ITEM, YFS_INVENTORY_DEMAND, and INV_INVENTORY_ITEM_LOCK tables cause severe contention and system degradation.

Hot SKU Properties (9.3)

PropertyPurposeDefault
yfs.inventory.sortandlock Sort and lock inventory records in a deterministic order to prevent deadlocks during concurrent updates false
yfs.balanceShortageAcrossNodes Balance backorder demand across all eligible nodes (vs. random assignment) false
yfs.nodecapacity.lock Use node capacity lock to serialize inventory updates at high-contention nodes false
persistCapacityAdjustments Persist capacity adjustments to additional tables for Hot SKU consolidation false
9.3 Hot SKU + Tag Controlled Items
Hot SKU cannot be enabled for tag-controlled items if Summarize and Maintain Total Demand and Supply Values flag is on. Turning it off leaves stale data in YFS_INVENTORY_TAG (TotalOnhandSupply, TotalOtherSupply, TotalDemand columns). Must run Inventory Purge Agent to clean up.

Availability Cache for High-Traffic Browsing

For product browsing pages (not checkout), use a short-lived availability cache. Instead of calling getATP for every page view, the cache stores a recent ATP snapshot per item-node. getAvailabilityCache reads from this in-memory store. Cache invalidation is driven by RTAM.

Safety Factor, Dirty Nodes & Node Controls

Safety Factor

The Safety Factor is a percentage of on-hand inventory excluded from promising — it acts as a buffer against forecast errors, in-transit losses, and store shrinkage.

Configuration
Safety factor is configured at item level in Catalog Management → Product → Node Item Attribute. You can set different safety factors per item-node combination (e.g., 5% safety factor at WH_EAST, 20% at Store_Mumbai_001 to account for higher store shrinkage).

Interview Cheat Sheet — Director / Principal Level

Top Clarifying Questions

  1. Are we discussing order-time ATP (live check during checkout) or channel-display ATP (RTAM cache)? The accuracy guarantees differ significantly.
  2. Is the vendor network INFINITE-node or TRACKED? This determines whether drop-ship demand appears in ATP calculations.
  3. What's the store inventory feed frequency — real-time WMS integration or batch delta? This determines how stale store ATP can be.
  4. Are there multi-enterprise (sub-division) configurations? ATP rules can differ per enterprise.
  5. Is the business using reservations at checkout or just OPEN_ORDER demand? Reservation mode affects concurrency and oversell risk.

Key Trade-offs to Verbalize

  • RTAM cache vs. live getATP: RTAM is fast but stale; live getATP is accurate but expensive at scale
  • Reservation vs. OPEN_ORDER: Mandatory reservation prevents oversell but increases latency at checkout; OPEN_ORDER is faster but risk of double-sell on flash sales
  • INFINITE vendor nodes: No oversell protection but maximum availability for drop-ship; TRACKED nodes are accurate but require EDI feed maintenance
  • Fine-grained DGs vs. coarse: More DGs = more granular sourcing control but higher RTAM computation load

Common Failure Modes

  • Backorder node-lock bug (9.3): Short-pick doesn't clear shipNode on order line → BACKORDER demand stays node-locked → RTAM publishes incorrect ATP → oversell
  • Unassigned demand inconsistency: getSupplyDetails sum ≠ getATP result → upstream system miscalculates ATP
  • Hot SKU DB contention: Flash sale without Hot SKU properties enabled → lock contention on YFS_INVENTORY_ITEM → timeout cascade
  • Schedule Agent frequency (OMS on Cloud 9.3): Minimum 30-min schedule frequency → backorders sit unprocessed for up to 30 minutes when new supply arrives
  • Safety factor misconfiguration: Wrong safety factor at item-node level → systematic over-promise or under-promise

"Wow" Moments — Depth Signals

  • Unassigned demand math: "I know that summing node-level getSupplyDetails will overstate enterprise ATP because unassigned demands (no shipNode) are excluded at node level — always use getATP with ConsiderUnassignedDemands=Y for enterprise truth."
  • RTAM + backorder distribution: "In 9.3, RTAM randomly assigns unassigned backorder demand to nodes for calculation purposes — this causes ghost ATP reductions at the wrong nodes. 9.5 introduced priority-based distribution. A workaround is ensuring backorder demands have shipNode cleared via changeShipment customization."
  • Inventory Considerations as config: "The supply types that count against a demand type are not hardcoded — they're configured in the Inventory Considerations matrix per DemandType. Misconfiguring ALLOCATED to see PO supply means you'd 'release' orders that can't physically ship yet."

Modern Inventory Availability — Target Architecture Principles

The replacement architecture applies five core patterns that collectively address all the constraints identified in Sterling GIV. These are not new ideas — they are proven patterns from companies like Amazon, Shopify, and Zalando who have built inventory systems at 10-100x Sterling's scale.

① Event Sourcing

Every inventory change is an immutable event. The current state is a projection. Replay enables any point-in-time snapshot.

② CQRS

Write path: inventory events → Kafka. Read path: pre-computed ATP per node/DG from Redis. Separate scaling for read-heavy ATP queries.

③ Microservices

Supply, Demand, ATP Calculation, Reservation, and Publish as independent services. Each owns its data store. Deploy, scale, and fail independently.

④ Eventual Consistency

Accept that display ATP (channel cache) and order-time ATP (live check) may differ by milliseconds. Design for it with optimistic locking at reservation.

⑤ API-First

All inventory capabilities exposed as versioned REST/gRPC APIs. No internal direct-DB access from consuming systems. Contract-first design.

Service Ownership & Boundaries

Supply Ledger Service
Replaces: YFS_INVENTORY_SUPPLY + adjustInventory + manageInventory

Owns all supply records. Accepts adjustments from WMS, ERP (PO updates), store feeds. Publishes supply events to Kafka. Source of truth for "what we have".

Demand Ledger Service
Replaces: YFS_INVENTORY_DEMAND + demand state transitions

Owns all demand records and their lifecycle (OPEN_ORDER → SCHEDULED → ALLOCATED → SHIPPED). State machine implemented as explicit transitions. Publishes demand events.

Reservation Service
Replaces: reserveAvailableInventory + cancelReservation

Handles atomic inventory reservation at checkout. Uses Redis DECRBY for lock-free atomic quantity reduction + writes RSRV_ORDER demand. TTL-based auto-release for abandoned carts.

ATP Calculator Service
Replaces: getATP + findInventory + Inventory Considerations Matrix

Consumes supply and demand events. Maintains pre-computed ATP per item-location in Redis sorted sets. Exposes live ATP API for order time checks. Handles node, group, and enterprise scopes.

ATP Publisher Service
Replaces: RTAM Agent + RTAM ATP publish

Consumes inventory.atp.updates topic. Pushes ATP snapshots to external channels via WebSocket, SSE, or REST callback. Replaces RTAM's agent-based polling model with reactive event-driven push.

Location Service
Replaces: Distribution Groups + ShipNode config

Manages locations (nodes) and location groups (DGs). Provides location metadata for ATP scope resolution. Config-driven — changes without code deployment.

Order Promising Service
Replaces: Schedule Agent + sourcing rules

Receives new orders. Calls ATP Calculator for availability. Applies sourcing rules (rule engine or simple config). Stamps ship node on order. Transitions OPEN_ORDER → SCHEDULED demand.

Replenishment Service
Replaces: Onhand Inventory Monitor + Action-Based Monitor

Subscribes to ATP update events. When ATP falls below configured threshold for an item-location, triggers replenishment workflow (Transfer Order creation, PO suggestion).

Dual-Write Bridge — Sterling → Modern

During co-existence, Sterling remains the write-master for inventory. The modern platform shadows Sterling data via two mechanisms:

CDC via Debezium (Supply)

Debezium captures row-level changes from Sterling's Oracle/DB2 database on YFS_INVENTORY_SUPPLY. Change events are published to Kafka topic sterling.cdc.supply. A transformer service converts Sterling's schema to the modern Supply Ledger format and writes to the new Supply Ledger Service.

Sterling DB → Debezium → Kafka:sterling.cdc.supply
  → Supply Transformer → Supply Ledger Service
  → PostgreSQL (new) + Kafka:inventory.supply.events

User Exit Bridge (Demand)

Sterling's User Exit framework is used to publish demand state transitions to Kafka. A lightweight UExit implementation on key APIs (createOrder, scheduleOrder, releaseOrder) publishes events to sterling.demand.events. Transformer maps to modern demand format.

Sterling UExit (createOrder/scheduleOrder)
  → Kafka: sterling.demand.events
  → Demand Transformer → Demand Ledger Service
  → PostgreSQL (new) + Kafka:inventory.demand.events

Shadow Mode Comparator

Before any production traffic is switched, the ATP Gateway runs in shadow mode for targeted items: it calls both Sterling's getATP AND the new ATP Calculator, compares results, and logs divergence. Sterling's answer is always served to the user. Divergence metrics are tracked on a Grafana dashboard. Only when divergence < 0.1% for 2 weeks does the feature flag flip to the modern service.

Architecture Decision Records

ADR-001 — Event Sourcing for Inventory Ledger over Mutable Records Accepted +

Sterling OMS uses mutable rows in YFS_INVENTORY_SUPPLY/DEMAND — a direct update model. This means the history is lost and audit trails require separate audit tables. We need to design the replacement data model.

Use an event-sourced append-only log as the write model. Every inventory change (adjustment, PO receipt, demand creation, demand transition) is recorded as an immutable event in PostgreSQL and simultaneously published to Kafka. The current state (ATP) is a projection from the event stream, not a mutable row.

✅ Full audit trail by design. ✅ Any point-in-time ATP snapshot is computable by replaying events. ✅ Kafka acts as the source of truth, enabling downstream projections. ❌ Query complexity — "what's the current inventory?" requires projection. Mitigated by pre-computed Redis cache.

Mutable update model (same as Sterling): rejected — loses history, same constraints we're migrating away from. Full event sourcing with event store (EventStoreDB): rejected — overkill, adds operational complexity; Kafka + Postgres provides 90% of the value at lower complexity.

ADR-002 — Redis as ATP Read Cache (CQRS Read Model) Accepted +

ATP queries are the highest-volume, most latency-sensitive operations — called at product browse time (millions of calls/hour) and at checkout (requires sub-10ms response). A full DB scan of supply and demand tables for every query is not viable at scale.

Maintain a pre-computed ATP value per item-location-group in Redis. The ATP Calculator Service consumes supply and demand events from Kafka and updates the Redis ATP hash in real time. All ATP queries read from Redis first; fall back to live DB calculation only on cache miss.

✅ Sub-millisecond ATP reads at any scale. ✅ Redis update lag is typically <100ms from inventory event to ATP update. ❌ ATP cache is slightly stale between event and Redis update — acceptable for display; live DB check used at order commit (reservation). ❌ Redis failover requires ATP recalculation from Kafka replay.

Materialized view in PostgreSQL: rejected — refresh latency of 1–5 seconds, lock issues during refresh under high write load. ElasticSearch ATP index: rejected — good for search but overkill for simple ATP key-value lookups; Redis sorted sets are more natural.

ADR-003 — Strangler Fig over Big-Bang Rewrite Accepted +

Sterling GIV processes millions of inventory events per day and is the availability backbone for all order promising. A single cutover event carries unacceptable business risk — one bug in the new system means all order promising fails.

Use Strangler Fig with feature-flag-based routing at the API Gateway. New system is built and validated in shadow mode. Traffic is migrated node-type by node-type (Warehouse → Store → Vendor) over 12 months. Sterling is always available as a rollback target during each wave.

✅ Zero-downtime migration. ✅ Rollback in <60 seconds at any phase. ✅ Shadow mode catches ATP divergences before real users are affected. ❌ Dual system operational cost for 15 months. ❌ CDC bridge adds complexity and is a temporary dependency. ❌ Team must maintain both systems in parallel.

Big-bang rewrite (all nodes at once): rejected — catastrophic blast radius. Any ATP calculation bug silently causes oversell or stock-out across all channels. Blue-green deployment (full parallel for one cutover): rejected — requires full feature parity before any production traffic; delays value delivery by 12+ months.

ADR-004 — Kafka Partition Key = item_id + location_id Accepted +

Inventory events for the same item at the same location must be processed in order — otherwise a later "decrease" could be applied before an earlier "increase", resulting in negative inventory. This is the distributed equivalent of Sterling's Hot SKU locking.

All Kafka topics use item_id:location_id as the partition key. All events for a given item-location always land on the same partition and are consumed by the same consumer instance — preserving order without any locking. This also collocates supply and demand events for the same item-location on the same partition, simplifying the ATP Calculator's join logic.

✅ No DB-level locking required for Hot SKU scenarios. ✅ Per-item-location ordering guarantee. ✅ Replaces Sterling's INV_INVENTORY_ITEM_LOCK table entirely. ❌ Hot partitions possible if a single item-location gets 100K events/sec during a flash sale. Mitigated by: (1) separate Kafka cluster/topic for Hot SKU items with higher partition count; (2) Redis atomic DECRBY for reservation (bypasses Kafka write for checkout).

Round-robin partitioning: rejected — no ordering guarantee, ATP Calculator needs distributed locking to compensate. Sharding by item_id only (not location_id): rejected — a single high-volume item across many locations would create a hot partition unnecessarily.

ADR-005 — Reservation via Redis Atomic Decrement (not DB lock) Accepted +

At checkout, multiple customers may attempt to reserve the last unit of a Hot SKU simultaneously. Sterling handles this with DB row-level locking on YFS_INVENTORY_ITEM — causing contention at scale. We need a reservation mechanism that handles 50K+ concurrent reservations/sec.

Reservation uses a two-step approach: (1) Redis DECRBY on the ATP counter — atomic, lock-free, ~200μs latency. If result ≥ 0, reservation succeeds. If < 0, INCRBY to revert and return "out of stock". (2) On success, asynchronously write RSRV_ORDER demand event to Kafka → Demand Ledger. If Kafka write fails, a saga compensates by INCRBY back in Redis.

✅ 50K+ reservations/sec without DB contention. ✅ Reservation latency <1ms (Redis) vs 10–50ms (DB lock). ✅ Eventual consistency with Kafka — acceptable because DB is not the source of truth anyway. ❌ Redis failure during window between DECRBY and Kafka write requires saga compensation. ❌ Redis cluster split-brain could temporarily allow double-reservation. Mitigated by Redis Cluster with async replication + Lua script for atomic reservation.

Postgres optimistic lock (version column): viable at <5K TPS but fails under flash sale load. DB SELECT FOR UPDATE (same as Sterling model): rejected — same contention problem being solved. Distributed lock (Redlock): rejected — adds latency and complexity; DECRBY is already atomic.
