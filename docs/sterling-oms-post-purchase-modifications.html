Executive Summary

IBM Sterling Order Management System (OMS) 9.3 is an enterprise-grade, rule-driven order orchestration platform built around a finite-state order pipeline. Post-purchase order modifications — Cancel, Address Update, Fulfillment Method Switch (Pickup ↔ Ship), and Service Level Change (1-day / 2-day / Standard 5-day) — are among the most operationally critical and technically complex flows in any retail deployment. Each modification must simultaneously satisfy order state eligibility checks, inventory re-reservation or release, payment recalculation and adjustment, carrier/node re-assignment, and downstream notification propagation.

⚠️ Critical Complexity Flag
Sterling's modification model is not a simple field-update. Every modification triggers a chain of API calls, condition tree evaluations, agent tasks, and pipeline transitions. Understanding the inner plumbing is essential before building any self-service or CSR modification UI on top of Sterling 9.3.
Core API Layer

All modifications flow through changeOrder API. Eligibility is gated by getOrderDetails + condition trees. Price changes use getPriceInfo + calculatePricing.

State Machine Gate

Sterling's order pipeline defines hard stops. An order past Sent to Node or in Shipment Confirmed status cannot be modified via standard APIs without a cancel+recreate pattern.

Inventory Impact

Address change + fulfillment switch can trigger full sourcing re-run (GIV/ATP re-evaluation), potentially reassigning fulfillment node, changing carrier, and altering promise dates.

Cutoff Time Logic — Critical for 1-Day

🚨 SLA Upgrade Cutoff Enforcement
Sterling does not natively enforce carrier cutoff times out of the box. You must implement a custom condition tree or pre-changeOrder user exit that: (1) calls getCapacityInfo for the current node with the requested carrier service, (2) checks if the current time is before the carrier cutoff (e.g., 2:00 PM for next-day air), (3) returns an error code SLA_UPGRADE_CUTOFF_MISSED if the window has passed, offering the next available option. Without this, customers can order "1-day" delivery at 11 PM and Sterling will accept it.

Architecture Decision Records

ADR-001 — Cancel+Recreate Pattern for Post-Node Fulfillment Switch Accepted +

Once an order reaches "Sent to Node" status, Sterling's standard changeOrder API cannot switch the fulfillment method (BOPIS↔Ship) because a physical or digital pick task has already been created at the node. Direct modification risks data inconsistency between the order and the node's work queue.

Implement a Cancel + Recreate pattern for fulfillment switches after node assignment: (1) Cancel the original order with reason FULFILLMENT_SWITCH, (2) Preserve order attributes (items, customer, promos), (3) Create a new order with the new fulfillment method, (4) Carry forward payment auth reference and void/re-authorize as needed. Original order number retained for CX continuity via a parent-child order relationship.

Pros: Clean state machine transition; no data inconsistency at node level; clear audit trail. Cons: Customer sees order number change (mitigated by showing original reference); double payment auth/void adds latency; promotions must be explicitly re-applied and may have expired.

Direct changeOrder force-modify: Rejected — bypasses pipeline guards, risks WMS task orphan records. Hold + manual reassign: Rejected — too slow for self-service CX; requires CSR intervention.

ADR-002 — Synchronous Payment Delta vs. Deferred Billing for SLA Upgrades Accepted +

When a customer upgrades from Standard to 1-Day shipping, there's a price delta that must be collected. The question is whether to charge it synchronously during the modification flow (blocking) or queue it for async collection via the Payment Collection Agent.

Synchronous auth + deferred capture: Perform an incremental authorization call to the PSP in real-time during the modification flow (via processPaymentConfirmation with ChargeType=ADDITIONAL_AUTH). If auth fails, block the modification. The actual capture is deferred to the existing payment capture pipeline (at shipment). This gives real-time feedback to the customer while not blocking on capture timing.

Pros: Customer knows immediately if payment fails; prevents service delivery without payment confirmation. Cons: Adds ~300–800ms latency to modification API call; PSP must support incremental auth (most modern PSPs do).

Fully async billing: Rejected — customer could receive 1-day delivery without successful payment, creating revenue leakage. Full synchronous capture: Rejected — capture before fulfillment is premature and complicates refund handling if order is subsequently cancelled.

ADR-003 — Idempotency Key for Modification API Calls Accepted +

Post-purchase modification APIs are called from customer-facing UIs over unreliable networks. A user hitting "Cancel" twice, or a mobile app retrying a failed SLA upgrade request, can result in double-cancel attempts or double-charges if the backend is not idempotent.

Implement an idempotency layer in the middleware/API gateway layer (not in Sterling itself, which lacks native idempotency keys). Each modification request is assigned a client-generated UUID as X-Idempotency-Key. The middleware layer stores (key → response) in Redis with a 24-hour TTL. On retry within TTL, the cached response is returned without re-invoking Sterling. Sterling's changeOrder API is called exactly once per logical modification.

Pros: Safe retries for all clients; prevents duplicate charges; aligned with PSP idempotency patterns. Cons: Requires Redis infrastructure; cache must be invalidated on order state changes that make the cached response stale.

Sterling-side duplicate detection: Considered but Sterling's duplicate check only works on order creation, not modification APIs. Database-level dedup: Too slow and complex to implement at the Sterling DB layer.

ADR-004 — Address Change: When to Trigger Full Re-Sourcing Accepted +

Not every address change warrants a full GIV re-sourcing run. A phone number change or apartment number change should be a simple field update. A ZIP code change across carrier zones should trigger full re-sourcing. The trigger rule needs to be precise to avoid unnecessary re-sourcing overhead.

Implement a re-sourcing trigger matrix in the beforeChangeOrder user exit: (1) Phone-only change → direct update, no re-source. (2) Address line / apartment change, same ZIP → update YFS_PERSON_INFO + update shipment address; no re-source. (3) ZIP change, same carrier zone → tax recalculation only. (4) ZIP change, different carrier zone → full GIV re-sourcing, carrier re-assignment, shipping cost recalc. (5) State change → full re-sourcing + tax re-calc. Zone lookup table maintained in configuration.

Pros: Avoids expensive GIV calls for trivial changes; reduces modification latency for 80% of address change cases. Cons: Zone table requires maintenance; edge cases (rural route delivery) may need manual override.

Always re-source on any address change: Rejected — creates 400–2000ms latency for every phone number update. Never re-source: Rejected — misses legitimate cost and fulfillment node changes.

Re-source Node Scoring Algorithm

/* Fulfillment Service — Node Scoring for Address Change Re-source */
/* Replaces Sterling's Sourcing Rules (distance rules + priority tiers) */

interface NodeScore {
  nodeId: string;
  score: number;  // 0–100 composite
}

function scoreNodes(items: LineItem[], destination: Address): NodeScore[] {
  return candidateNodes.map(node => {
    const atp        = inventoryService.getATP(node, items);       // 0 or 1
    const distance   = geoService.calcZones(node.zip, destination.zip);
    const transitDays = carrierService.getTransitDays(node, destination, serviceCode);
    const cost       = pricingService.getShippingRate(node, destination, serviceCode);
    const capacity   = fulfillmentService.getNodeCapacity(node);   // 0–1

    // Weighted scoring — tune per retailer business rules
    return {
      nodeId: node.id,
      score: (atp * 40)              // must have stock — highest weight
           + ((10 - distance) * 20)  // closer is better
           + ((7 - transitDays) * 20)// faster delivery
           + ((20 - cost) * 10)      // cheaper shipping
           + (capacity * 10)         // node not overloaded
    };
  }).filter(n => n.score > 0)        // only nodes with inventory
    .sort((a, b) => b.score - a.score);
}

Migration Architecture Decision Records

ADR-M01 — Strangler Fig over Big Bang Rewrite Accepted +

Sterling OMS processes live order modifications 24/7. A big-bang cutover to a new system would require a maintenance window of days and risks catastrophic failure with no proven rollback path for an order management system handling real revenue.

Strangler Fig with an API Gateway routing proxy. Sterling remains the system of record for Waves 1–3. New services are introduced incrementally via feature flags. Each domain can roll back independently in under 2 minutes by flipping a feature flag. Sterling is decommissioned only after all domains are live and stable for 30 days.

Pros: Zero-downtime migration; incremental risk; independent domain rollback; real traffic validation on each service before full cutover. Cons: 12–18 month migration timeline vs. 6 months for big-bang; dual system maintenance cost during transition; engineers must understand both Sterling and new services simultaneously.

Big Bang Rewrite: 6-month build + cutover weekend — rejected due to catastrophic rollback risk and promotion/pricing regression risk on live orders. Lift-and-shift v9.3 to containers: Sterling 9.3 is not container-native (EAR-based WebSphere); containerization only available in v10+; rejected as it preserves all architectural debt.

ADR-M02 — CQRS + Event Sourcing for Order Service Accepted +

Sterling's order modification history is stored in YFS_ORDER_AUDIT as snapshots. There's no replay capability. The new Order Service needs a complete, queryable modification history, support for event replay during migration, and audit compliance for financial regulations.

Implement CQRS on the Order Service: Write side = command handlers that produce events stored in a Kafka-backed event log (with PostgreSQL event store for durability). Read side = projections materialized into PostgreSQL (operational queries) and Elasticsearch (historical search). Every modification is an immutable event: OrderCancelled, OrderAddressChanged, etc.

Pros: Complete audit trail; event replay for migration verification; temporal queries ("what was the order state at time T?"); decoupled read/write scaling. Cons: Eventual consistency on read models (acceptable for order history); engineering complexity (2x vs. simple CRUD); team needs CQRS/ES training runway.

Simple CRUD with audit log table: Rejected — loses modification intent semantics; can't replay events to rebuild read models; doesn't support the migration verification requirement.

ADR-M03 — Debezium CDC over Custom Dual-Write Application Logic Accepted +

During migration, changes in Sterling's Oracle DB must be propagated to new services' PostgreSQL databases and Kafka. Two options: (1) modify Sterling application code to dual-write to both DBs, or (2) use Debezium CDC to tap Oracle's redo log.

Debezium CDC (Oracle connector) reading from Oracle redo log → Kafka topics. This requires zero changes to Sterling application code, captures every DB change including those from Sterling's internal agents, and provides a reliable event stream even if the application layer crashes mid-transaction. Debezium's Oracle connector requires Oracle LogMiner configuration.

Pros: No Sterling code changes; captures ALL mutations (including agent-driven changes not triggered by APIs); low latency (sub-second); proven at scale. Cons: Oracle LogMiner licensing implications — verify with Oracle; Debezium Oracle connector has schema change complexity; requires DBA involvement for LogMiner setup.

Application-level dual-write in Sterling Java user exits: Rejected — requires Sterling code changes; only captures API-driven mutations, misses agent-driven mutations; hard to maintain during migration.

ADR-M04 — Inventory Service: Go over Java Spring Boot Accepted +

The Inventory Service must handle concurrent ATP reservation requests at high throughput during flash sales (up to 50K RPS). Reservation contention — multiple threads competing to reserve the last unit at a node — is the critical performance bottleneck. Language choice matters for this service more than any other.

Go (Golang) for the Inventory Service. Go's goroutine model handles 50K+ concurrent connections with minimal memory overhead vs. JVM thread pools. Go's native mutex primitives map cleanly to reservation lock patterns. Binary size and startup time are also better for horizontal autoscaling during peak traffic bursts.

Pros: 5–10x better concurrency handling vs. JVM thread-per-request; lower memory footprint (50MB vs. 500MB JVM heap); fast startup (autoscale in <2s). Cons: Most retail engineering teams are Java/Node.js dominant — Go is a hiring and training investment; Go's ORM ecosystem is less mature than Spring Data JPA.

Java Spring Boot: Adequate but JVM thread pools become a bottleneck at 20K+ concurrent reservations without reactive programming. Node.js: Single-threaded — cannot use multi-core CPU for CPU-bound ATP calculations.

Migration Interview Cheat Sheet — Director Level

🎯 Top Questions Before Starting Migration

  • What is the migration trigger — IBM license renewal date, performance cliff, or developer velocity problem? The answer shapes urgency and risk tolerance.
  • Does the business have complex omnichannel fulfillment rules (BOPIS, SFS, curbside)? If yes, strongly consider Fluent Commerce over a full custom build.
  • What's the team size? A team of 5 should not attempt a full custom OMS — buy a SaaS OMS and customize APIs.
  • What peak traffic does the business need to handle? If it's <5K RPS and no flash sales, Sterling's limits aren't the real problem.
  • Is there a 2-year runway or a 6-month runway? This determines whether Strangler Fig or a parallel-run big-bang is more realistic.

⚡ Key Trade-offs to Verbalize

  • Strangler Fig timeline vs. big bang risk: 18 months of dual-system cost vs. catastrophic weekend cutover risk
  • Build vs. Buy sourcing engine: Sterling's sourcing engine took IBM 20 years to build — rebuild that vs. license Fluent Commerce
  • Eventual consistency in sagas: Cloud-native modifications are async sagas; Sterling was synchronous. CX implications of "pending" states
  • CQRS complexity vs. audit benefit: Event sourcing on Order Service adds engineering overhead but gives migration verification and compliance audit for free

🔍 Hardest Technical Problems

  • Promotion parity: Sterling's promotion engine encodes years of business rules in Condition Trees and Java user exits. Getting 100% pricing parity in a new engine is the #1 migration defect source.
  • Inventory ATP accuracy during dual-write: The 4-week window where both Oracle and PostgreSQL are accepting reservations requires a bulletproof reconciliation script running every hour, not nightly.
  • Event ordering guarantees: Kafka ordering within a partition but cross-service sagas can arrive out of order — use event sequencing + idempotent consumers.
  • Historical order data: 5+ years of orders in Sterling Oracle — you cannot migrate all to PostgreSQL; archive to S3 Parquet and provide a separate historical query service.

💡 Director-Level "Wow" Signals

  • Seasonal awareness: "I'd schedule the inventory cutover for February — after holiday peak and before spring sale season. The Gantt should have hard migration freezes around Black Friday, Cyber Monday, and Q4 peak."
