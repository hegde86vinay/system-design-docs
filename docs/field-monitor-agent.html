📋Project Summary

Field Monitor is a daily, single-purpose agent that produces a personal news digest of Software Architecture content sourced exclusively from Medium. It runs headless on macOS via launchd, uses a persistent Playwright browser session to read member-only Medium tag pages, deduplicates and ranks the last 24 hours of articles, and delivers a single self-contained HTML file.

Per-article cards show title, author, read time, claps, and the snippet Medium publishes on the page — no LLM summarization. The report ends with a styled summary table (# | Topic | Article Title | Author | URL) so the reader can scan everything at a glance and click through.

🎯Goals & Constraints

Goals

  • Strictly Medium-only. No cross-source aggregation, no fallback to RSS aggregators or news APIs.
  • Premium-aware. Authenticate as a Medium member so member-only and personalized topic feeds are visible.
  • Five fixed topics (no dynamic topic selection or LLM-driven taxonomy).
  • Last 24 hours only. Top 5 per topic by claps, after global URL dedup across topics.
  • Single self-contained HTML for the report — no external CSS/JS, no build step, opens in any browser.
  • Final markdown-style table with #, Topic, Article Title, Author, URL.
  • Truly daily. Survives Mac sleep — cron-style "skip the day if asleep" is unacceptable.
  • Zero ongoing cost. No LLM bill, no cloud bill, no SaaS bill.

Non-goals

  • Per-article LLM summaries (would add cost; deferred until needed).
  • Publishing back to Medium as a draft post (read-only consumer, not producer).
  • Multi-user — single user, single laptop.
  • "Trending" or social-graph ranking — claps in the last 24h is good enough.
Why no LLM summaries? The Medium card snippet is usually the lede sentence — already the author's chosen hook. Adding a model in the loop introduces cost, latency, and quality variance for marginal benefit. The architecture supports adding summaries later (see If You Add Summaries).

🔑① Auth — Playwright Persistent Context

auth.py wraps chromium.launch_persistent_context(user_data_dir=.auth/chromium-profile). The profile dir holds cookies, localStorage, and IndexedDB across runs, so a single interactive Google OAuth (run once via --first-login --headed) keeps the Medium session warm for weeks.

The login flow is intentionally manual: Google's bot defenses make scripted credential entry from Playwright unreliable, and credentialing through MFA / device-trust prompts cannot be automated without a sanctioned API. The agent pauses for a human to complete the flow once, then runs headless thereafter.

Each run starts with a cheap probe (GET /me) to confirm the session is still alive. A failed probe sends a short "[Field Monitor] Auth required" email and exits — the next run won't fix itself.

🕷️② Fetcher — Per-Tag Scrape

fetcher.py visits https://medium.com/tag/{slug}/recent, scrolls four times to trigger lazy-loading (~40 cards), then parses <article> elements with BeautifulSoup. Each card yields:

  • Title from the first h2/h3
  • URL from the first non-author anchor (strips query/source params)
  • Author from a[rel=author] or data-testid="authorName"
  • Snippet from the first p (truncated to 280 chars)
  • Read time via regex on the card text ("\d+ min read")
  • Claps from the clap button's aria-label (parses 1.2K / 3,420)
  • Published at from <time datetime=...>, normalized to UTC

Articles older than the 24-hour window are dropped. Between tag fetches the fetcher sleeps random.uniform(3, 6) seconds. Cloudflare interstitials (detected via title "Just a moment...") trigger one 60-second back-off and retry, then the tag is skipped — never the whole run.

🔀③ Pipeline — Fan-out, Dedup, Top-N

pipeline.py opens the persistent context once, runs the auth probe, then walks TOPIC_TAG_MAP in order. For each topic it concatenates results across all of that topic's tags into a per-topic list, with a single global seen_urls set that prevents the same article appearing under two topics (an article tagged both aws and cloud-computing lands wherever it's seen first).

Within a topic, articles sort by claps DESC and the top TOP_N_PER_TOPIC = 5 survive. Empty topics are kept in the result dict (the renderer shows a placeholder card) so the layout never collapses.

🎨④ Render — Pure-Python f-string Templating

render.py emits a single ~30 KB HTML file. CSS, JavaScript, and content are inlined — the file is portable and survives email forwarding. The template reuses the GitHub-dark CSS variable palette from this docs collection, so the generated digest visually matches the system-design docs you're reading now.

Structure:

  • Fixed-position sidebar with topic anchors + IntersectionObserver-based active highlighting
  • Hero with date, total article count, populated-topic count
  • One <section> per topic, with cards (title, meta, snippet, "Read on Medium →")
  • A final <table class="summary"> styled to look like a markdown grid: # | Topic | Article Title | Author | URL

No Jinja2 — the templating volume doesn't justify the dependency, and the existing projects in this repo all use the same plain-string convention.

📧⑤ Emailer — Gmail SMTP_SSL

emailer.py builds a MIMEMultipart("alternative") with the full HTML as text/html and a plain-text summary fallback. The 16-character Gmail App Password is fetched from Keychain at send time. SMTP failures retry twice with 30 s backoff; final failure logs to logs/email-failures.log and exits non-zero so launchd surfaces the error in .err.

The HTML is emailed in the body, not as an attachment, so phone clients render it inline. The same file is already on disk in reports/ regardless of email outcome.

⏰⑥ Scheduler — macOS launchd

Unit file at ~/Library/LaunchAgents/com.vinayhegde.field-monitor.plist with StartCalendarInterval set to {Hour: 20, Minute: 0} and RunAtLoad: false. Standard out/err redirect to logs/field-monitor.{out,err}.

🎚️Dedup & Ranking

  • Dedup is global, not per-topic. An article tagged aws and cloud-computing shows up once, in whichever topic was processed first (TOPIC_TAG_MAP insertion order, which is the canonical topic order).
  • Ranking is claps DESC within topic. Claps are a noisy proxy but cheaper to extract than reading time × engagement, and the 24h window already filters for "happening now".
  • Top-N is 5 per topic. A digest with ≤25 articles fits one screen on a laptop and one phone screen of scrolling.

🛡️Anti-Bot & Terms-of-Service Posture

This agent is a single-user, low-volume scraper that visits ~17 public-or-member-visible pages once per day. That's well below normal interactive browsing. The defensive layers:

  • Persistent context reuses cookies / localStorage / IndexedDB so traffic looks like the same logged-in user across runs, not a fresh fingerprint.
  • Realistic Chrome user-agent matching the persistent profile's actual Chromium build.
  • Scroll & pause — 4 scroll passes with 1.5 s pauses, plus uniform(3, 6) s jitter between tags.
  • Single concurrent connection — no parallel tag fetches.
  • Cloudflare detection — back off and skip rather than hammer.

playwright-stealth is installed (patches navigator.webdriver and other tells) but is not strictly required at this volume.

Be a good citizen. Medium's ToS permits personal-use viewing of member content. Do not redistribute scraped article bodies or snippets publicly. The digest is for personal consumption and is sent only to the operator's own email.

🟧AWS Bedrock

What Bedrock actually is: a managed inference API for foundation models (Anthropic Claude, Amazon Nova, Meta Llama, Mistral, Cohere, etc.) accessed via the AWS SDK with IAM-based auth.

Bedrock is not a hosting platform. It will not run your Playwright scraper, render HTML, or send email. You still need Lambda / Fargate / EC2 around it. Bedrock is only relevant if you upgrade Field Monitor to use an LLM (e.g. for per-article summaries or topic re-classification).

When Bedrock is the right LLM choice

  • You're already on AWS and your org requires that all data stay inside an AWS account / VPC for compliance reasons.
  • You want IAM-managed access keys instead of a separate Anthropic API key.
  • You'll consolidate the bill on AWS.

🐳ECS Fargate Scheduled Task

Heavier than Lambda but no cold start, no 15-min execution cap, and full control over the container. Best when you might extend the agent later (more topics, longer scrapes).

  • 0.25 vCPU, 0.5 GB RAM, 5 min/day → ~$0.05–0.10/mo for compute
  • For Playwright + Chromium realistically need 0.5 vCPU, 2 GB → ~$0.25/mo
  • Add EFS, SES, EventBridge → total ~$0.50/mo

☁️GCP Cloud Run Jobs + Cloud Scheduler

GCP's closest analogue to "scheduled container task". Cloud Run Jobs handles short-lived containers; Cloud Scheduler triggers them via HTTP or Pub/Sub.

  • 1 vCPU × 2 GB × 5 min × 30 = ~$0.10–0.30/mo compute
  • Cloud Scheduler: 3 jobs free / month forever
  • Secret Manager: ~$0.06/mo per active secret
  • SendGrid free tier (100 emails/day) for delivery
  • Total: ~$0.30/mo

Notable: GCP Cloud Run recently added persistent volume support, which simplifies the Playwright cookie-persistence problem vs. Lambda.

🟦Azure Container Apps Jobs

Azure's scheduled-job offering. Triggered by Azure Logic Apps or a cron schedule directly inside Container Apps Jobs.

  • ~$0.20–0.40/mo for the same Playwright workload
  • Key Vault ($0.03 per 10K operations)
  • Communication Services Email (per-message billing)
  • Total: ~$0.40/mo

🐙GitHub Actions (cheapest cloud option)

If you're willing to commit the Field Monitor source to a private GitHub repo, you can run the daily job on GitHub-hosted runners under the free tier. This is the lowest-friction cloud path.

  • Free tier: 2,000 min/mo on private repos. Field Monitor uses ~5 min/day = 150 min/mo — well under.
  • Public repo: unlimited free minutes.
  • Trigger: schedule: - cron: '0 14 * * *' (14:00 UTC = 20:00 local for IST/Asia).
  • Secrets: GitHub Encrypted Secrets — encrypted at rest, exposed as env vars to the job.
  • Storage: commit the Playwright storage_state.json as a base64-encoded GitHub Encrypted Secret, or upload to S3 / R2.

🧩Self-hosted n8n / Activepieces / Make

If you already operate a home server (Synology, Raspberry Pi, NAS, mini-PC), running n8n locally gives you a visual workflow editor with a built-in scheduler. The Field Monitor pipeline becomes 4–6 nodes: HTTP Request → HTML Extract → Function (dedup) → HTML render → Send Email.

  • Cost: $0 (existing hardware) or ~$5/mo for the n8n Cloud Starter plan.
  • Pro: No code to maintain. Easy to extend with more sources later.
  • Con: Playwright is harder to wire into n8n than into a Python script. Less idiomatic for headless-browser scraping.

📖Run Book

  1. One-time setup

    python3 -m venv .venv && source .venv/bin/activate && pip install -r requirements.txt && python -m playwright install chromium

  2. Stash secrets

    cp .env.example .env && python scripts/store_secrets.py — interactive, writes to Keychain.

  3. First Medium login

    python main.py --first-login --headed. Click "Sign in with Google", complete MFA, return to terminal, press Enter.

  4. Verify dry run

    python main.py --dry-run --open-browser opens today's HTML in your default browser. No email is sent.

  5. Install schedule

    cp scripts/com.vinayhegde.field-monitor.plist ~/Library/LaunchAgents/ && launchctl load -w ~/Library/LaunchAgents/com.vinayhegde.field-monitor.plist

  6. Force-fire to validate

    launchctl start com.vinayhegde.field-monitor && tail -f logs/field-monitor.out logs/field-monitor.err

⚠️Known Limitations

  • Mac must be on at 20:00 (or wake within the day). launchd catches up on next wake but the digest will be stale-by-N-hours. If the Mac is off for 36+ hours the run is missed entirely.
  • Login session expires every few weeks. Google's session lifetime is opaque — when it lapses, the agent emails an alert and exits. Re-running --first-login --headed is a 30-second fix.
  • Cloudflare is the silent killer. Most failures we'll see are Cloudflare interstitials on individual tag pages, not bugs in the scraper. The partial-success policy hides them in the digest with a yellow banner.
  • Claps as ranking signal. Recent articles haven't accumulated claps yet, so the freshest content can be ranked below 12-hour-old viral pieces. Not great, but no obviously better signal is available without a model.
  • Snippet quality varies. Some Medium authors don't write a strong lede; their snippets read as truncated mid-sentence. Adding LLM summaries fixes this — at the cost analyzed above.
