Overview — The Agent Processing Model

IBM Sterling OMS uses an agent-based asynchronous processing model to execute the vast majority of its order lifecycle logic. Rather than performing long-running operations synchronously in the API layer, Sterling delegates them to background agents that poll for work, process it, and advance order state.

This design is core to Sterling's architecture and directly shapes how you tune performance, troubleshoot stuck orders, and plan for modernization. Every engineer working with Sterling must deeply understand these three processing constructs:

⚡ Task Queue (Q) Agent

Event-driven. Polls the YFS_TASK_Q table. Fires when a triggering condition (status change, event) deposits a row. The workhorse of order state transitions.

🕐 Time-Triggered Agent

Cron-style. Runs on a fixed schedule regardless of demand. Used for batch operations, SLA monitoring, inventory synchronization, and housekeeping.

🔌 Integration Server

Message-bus bridge. Consumes from JMS/MQ queues and dispatches inbound XML payloads to Sterling APIs. Also publishes outbound events to external queues.

🧩 Non-Task Q Agent

Custom batch agent using getJobs() + executeJob(). Extends YCPBaseAgent. Works on records not tied to order/shipment entities — e.g. custom table purges, external data sync.

💡 Why This Matters for GCC Interviews
At director/principal level, you're expected to explain why orders get stuck, how to tune throughput for peak sale events, and how agents map to Kubernetes workloads in v10.x. These three constructs are the answer to all three questions.

Task Queue (Q) Based Agents

Core Concept
Task Q agents are event-driven agents that react to order lifecycle events. When Sterling's event framework fires an event (e.g., ON_SUCCESS of order creation), it writes a row into YFS_TASK_Q. Task Q agents poll this table, pick up rows, and execute the configured action (typically invoking an API or custom Java class).

⚠️ The "INPROCESS Stuck" Problem
If an Agent JVM crashes mid-processing, rows remain in INPROCESS state indefinitely. Sterling provides a Hung Task Recovery Agent (a time-triggered agent) that reclaims rows stuck in INPROCESS beyond a configurable timeout. Always configure this agent in production.

Time-Triggered Agents

Core Concept
Time-triggered agents run on a fixed schedule (cron-like). They do not react to events — they scan the database on their own cadence and process whatever they find. They're used for batch operations, SLA enforcement, inventory sweeps, and purges.

Overlap Risk: Time-Triggered + Task Q Contention

Both agent types share the same thread pool if co-deployed in one JVM. A heavy batch run of the Purge Agent at the wrong time can starve Task Q agents processing live orders. Best practices:

  • Schedule purge/housekeeping agents in off-peak windows (midnight–4AM)
  • Set MaxRecordsToBuffer conservatively for purge agents (500 max) to limit DB lock contention
  • In v10.x, run purge workloads as separate Kubernetes CronJob pods, isolated from the main om-agent deployment
  • Monitor thread pool utilization via JMX / Prometheus — alert at >80% saturation

Integration Server

Core Concept
The Integration Server is Sterling's asynchronous message-bus bridge. It listens on JMS queues (IBM MQ, ActiveMQ), receives inbound XML messages from external systems (ERP, WMS, store POS), transforms/routes them to Sterling Java APIs, and publishes outbound events back to queues for downstream consumers.

Inbound vs. Outbound Processing

📥 Inbound (External → OMS)

  • IS listens on one or more JMS queues
  • Each queue mapped to a service/API name
  • Optional XSL transform to normalize external XML to YFS format
  • Message dispatched to Sterling API (callMultiApi)
  • On success: message ACK'd and removed from queue
  • On failure: retry up to N times → dead letter queue
  • Transactional: LOCAL (per-message) or XA (distributed)

📤 Outbound (OMS → External)

  • Sterling event framework fires on order state changes
  • Event subscription configured to publish to JMS queue
  • IS (or event handler) serializes order XML and posts to queue
  • Downstream consumers (Kafka Connect, notification service) pick up
  • Common outputs: shipment confirmation, inventory change, invoice
  • Can route to multiple queues based on enterprise/document type

Integration Server vs. Modern Event Streaming

Architectural Tension in Modernization
The Integration Server was designed for JMS/MQ in an era before Kafka. In modern GCC architectures, teams often bridge Sterling's IS to a Kafka Connect MQ Source Connector: IS still talks to MQ, and a Kafka Connect worker replicates MQ messages to Kafka topics consumed by microservices. This avoids rewriting Sterling internals while gaining Kafka's replay, consumer groups, and schema registry benefits.
DimensionIntegration Server (JMS/MQ)Modern: Kafka Connect Bridge

Replay / rewind✗ Messages consumed = gone✓ Kafka retention window
Multiple consumersLimited (competing consumers)✓ Consumer groups, fan-out
Schema evolutionManual XSL versioning✓ Schema Registry + Avro
Operational toolingMQ Explorer / log files✓ Kafka UI, KSQL, Grafana
Sterling code changeZero — IS unchangedZero — bridge is external

Dead Letter Queue (DLQ) Handling

Every production IS deployment must have a DLQ strategy. When a message fails all retries:

  • Message lands in configured error queue (e.g. STERLING.ERROR.DLQ)
  • Build a DLQ monitor: alert on non-zero DLQ depth (PagerDuty / Prometheus)
  • Include correlation IDs and original payload in DLQ message for reprocessing
  • Provide an ops runbook for DLQ reprocessing (fix root cause → replay)
  • Never silently swallow exceptions — missing the DLQ is how orders disappear

Non-Task Q Based Agents

Core Concept
A Non-Task Q agent is a custom batch agent that does not rely on the YFS_TASK_Q table. Instead, it implements two Java methods — getJobs() and executeJob() — by extending Sterling's YCPBaseAgent class. You define what records to fetch and how to process each one. This pattern is ideal for processing records that have no relationship to orders or shipments, such as custom table purges or external data synchronization.

According to the IBM Sterling Agent Architecture document (Sandeep Khare, IBM India, Oct 2011), the underlying mechanism for all Sterling time-triggered agents — including Non-Task Q custom agents — is the same JMS-backed getJobs/executeJob pipeline. The key distinction is that Task Q agents use a fixed OOB implementation of getJobs (querying YFS_TASK_Q), whereas Non-Task Q agents let you supply your own getJobs implementation to query any table or data source.

The Two Key Methods

getJobs(YFSEnvironment env, Document criteria, Document lastMessageCreated)

  • Always single-threaded — enforced via YFS_OBJECT_LOCK table lock per AgentCriteriaId
  • Receives agent criteria parameters as XML input (org code, date range, etc.)
  • Queries the data source and returns a List<Document> — one Document per record to process
  • lastMessageCreated is null on first invocation; subsequent calls carry the last processed key for cursor-style pagination
  • Each Document in the returned list is posted as an execute message to the internal JMS queue
  • You control batch size via NumRecordsToBuffer (default: 5000)

executeJob(Document job)

  • Multi-threaded — runs concurrently with N threads (configurable)
  • Called once per Document returned by getJobs
  • Contains the business logic: call a Sterling API, delete a record, call an external service
  • Each thread picks one execute message from the JMS queue independently
  • Multiple JVM instances can run executeJob in parallel — getJobs locking ensures no duplicate fetches
  • Failures per record should be handled inside this method (try/catch + log)

⚠️ Important: getJobs Is Always Single-Threaded
Even in multi-JVM deployments, only one JVM's getJobs() runs at a time — controlled by a lock on YFS_OBJECT_LOCK for the agent's CriteriaId. If a second JVM's getJobs() cannot acquire the lock, it exits silently. This prevents the same batch of records being fetched twice. executeJob() is where parallelism lives.

💡 Agent Will Not Re-trigger While Processing
Per IBM documentation, an agent will not accept a new trigger (manual or automatic) while it is actively processing records. A manual trigger issued during processing is silently ignored. Auto-trigger will not fire until the current batch cycle is fully complete.

Custom Table Requirements

When building a Non-Task Q agent for a custom table, the table must support the agent framework's pagination contract:

  • A unique primary key column (UUID or sequence) that can serve as the lastMessageCreated cursor
  • Index on the columns used in the getJobs() SELECT criteria (e.g. CREATE_TS for purge, STATUS for processing queues)
  • Ensure the query used in getJobs() is deterministically ordered — typically ORDER BY PRIMARY_KEY ASC — so cursor pagination is correct across batches

Key Design Decisions for v10.x Agent Deployment

ADR-001 — Separate IS from Task Q agents into distinct Deployments Accepted +

In v9.x, all agent types share one JVM. In v10.x we have the opportunity to separate concerns at the pod level.

Run Integration Server listeners in a dedicated om-integration-server Deployment. Run Task Q + time-triggered agents in om-agent Deployment. Run purge/housekeeping as Kubernetes CronJobs.

✅ IS can scale independently based on MQ queue depth. ✅ Purge jobs don't compete for threads with live order processing. ✅ Easier to tune resource requests/limits per workload type. ⚠️ More Helm chart complexity; need separate ConfigMaps per deployment.

Single om-agent Deployment for all — rejected due to thread pool contention and inability to scale IS independently.

ADR-002 — HPA on custom metric: YFS_TASK_Q OPEN row count Accepted +

Default CPU-based HPA lags behind actual agent demand. Queue depth is a more direct signal.

Expose YFS_TASK_Q OPEN count as a custom metric via a Prometheus exporter (SQL query → Prometheus gauge). Configure HPA with externalMetrics pointing to this metric. Scale out when queue depth > 200 rows.

✅ Proactive scaling — pods added before queue depth causes latency. ✅ Scales down during off-peak to save cost. ⚠️ Custom metric exporter adds operational complexity; needs its own monitoring.

CPU-based HPA — lags 2–3 minutes behind actual load spikes, causing order processing latency during flash sales.

Custom Agent Framework — Architecture Design

Design Challenge
If you were to rebuild Sterling's four agent types outside of Sterling — in a cloud-native, language-agnostic custom framework — what would the architecture look like? This is a principal-level system design question. The goal is to preserve the behavioral contracts of each agent type while gaining observability, horizontal scalability, and cloud-native operability that Sterling's legacy JVM model lacks.

Design Principles

  • Behavioral parity: Each agent type must retain its core contract — event-driven, cron, message-bridge, or custom batch.
  • Separation of concerns: Trigger, fetch, dispatch, and execute are distinct lifecycle stages — each independently scalable.
  • Idempotency by default: Every agent must be safe to re-run on the same record without side effects.
  • Observability first: Queue depth, processing latency, error rate, and retry counts are first-class metrics — not afterthoughts.
  • Cloud-native deployment: Stateless workers, Kubernetes-native autoscaling, and configuration via environment variables / ConfigMaps.

① Task Q Worker — Kafka + KEDA Autoscaling

// TypeScript / Node.js — Kafka Consumer Worker (Task Q equivalent)
import { Kafka, EachMessagePayload } from 'kafkajs';
import Redis from 'ioredis';

const kafka = new Kafka({ brokers: ['kafka:9092'] });
const consumer = kafka.consumer({ groupId: 'schedule-order-workers' });
const redis = new Redis({ host: 'redis' });

await consumer.connect();
await consumer.subscribe({ topic: 'order.state.events', fromBeginning: false });

await consumer.run({
  partitionsConsumedConcurrently: 3,   // matches thread count concept
  eachMessage: async ({ topic, partition, message }: EachMessagePayload) => {
    const event = JSON.parse(message.value!.toString());

    // Idempotency check (equivalent to INPROCESS guard in YFS_TASK_Q)
    const idempKey = `processed:${topic}:${partition}:${message.offset}`;
    const alreadyDone = await redis.set(idempKey, '1', 'EX', 86400, 'NX');
    if (!alreadyDone) return;  // duplicate — skip, offset will be committed

    try {
      await scheduleOrder(event.orderId, event.payload);
      // Offset committed automatically on next poll — equivalent to STATUS=COMPLETED
    } catch (err) {
      await redis.del(idempKey);  // release so retry can re-acquire
      throw err;  // triggers KafkaJS retry → eventually DLT
    }
  }
});

// Purge agent implementation (Time-Triggered Agent equivalent)
async function runPurgeAgent() {
  const retentionCutoff = subDays(new Date(), parseInt(process.env.RETENTION_DAYS!));
  const batchSize = parseInt(process.env.BATCH_SIZE!);
  let cursor: string | null = null;
  let totalProcessed = 0;

  do {
    // Equivalent to getJobs() — fetch a page of records
    const records = await db.query(
      `SELECT id, order_key FROM order_archive
       WHERE created_at < $1 AND ($2::text IS NULL OR id > $2)
       ORDER BY id ASC LIMIT $3`,
      [retentionCutoff, cursor, batchSize]
    );

    if (records.length === 0) break;

    // Parallel executeJob equivalent
    await Promise.allSettled(
      records.map(r => deleteOrderRecord(r.id).catch(e => {
        logger.error({ id: r.id, err: e.message }, 'purge record failed');
        metrics.increment('purge.record.error');
      }))
    );

    cursor = records[records.length - 1].id;
    totalProcessed += records.length;
    metrics.gauge('purge.records_processed', totalProcessed);
  } while (true);

  logger.info({ totalProcessed }, 'purge cycle complete');
}

// Router Service — stateless pod consuming from Kafka inbound topics
// Equivalent to Sterling Integration Server message routing

import { Kafka } from 'kafkajs';
import Ajv from 'ajv';
import { inventorySchema, shipmentSchema } from './schemas';

const TOPIC_ROUTING: Record<string, { schema: object; handler: Function }> = {
  'inbound.inventory.updates':        { schema: inventorySchema,  handler: manageInventory },
  'inbound.shipment.confirmations':   { schema: shipmentSchema,   handler: confirmShipment },
};

await consumer.run({
  eachMessage: async ({ topic, message }) => {
    const routing = TOPIC_ROUTING[topic];
    if (!routing) { logger.warn({ topic }, 'No routing config'); return; }

    let payload: unknown;
    try {
      payload = JSON.parse(message.value!.toString());
    } catch {
      await publishToDLT(topic, message, 'PARSE_ERROR');  // equivalent to Sterling DLQ
      return;
    }

    // Schema validation (equivalent to XSL validation in Sterling IS)
    const ajv = new Ajv();
    if (!ajv.validate(routing.schema, payload)) {
      await publishToDLT(topic, message, `SCHEMA_INVALID: ${ajv.errorsText()}`);
      return;
    }

    // Dispatch to domain service (equivalent to callMultiApi in Sterling IS)
    try {
      await routing.handler(payload);
    } catch (err: any) {
      // Retry handled by Kafka consumer retry topics
      throw err;
    }
  }
});

④ Non-Task Q Agent — Temporal Workflow

// Temporal Workflow — Non-Task Q Agent equivalent
// workflow.ts
import { proxyActivities, sleep, condition } from '@temporalio/workflow';
import type * as activities from './activities';

const { getJobsActivity, executeJobActivity } = proxyActivities<typeof activities>({
  startToCloseTimeout: '10 minutes',
  retry: { maximumAttempts: 3, initialInterval: '5s', backoffCoefficient: 2 }
});

export interface BatchWorkflowInput {
  criteria: Record<string, unknown>;
  batchSize: number;
}

// Equivalent to the full getJobs → executeJob cycle
export async function batchPurgeWorkflow(input: BatchWorkflowInput): Promise<void> {
  let cursor: string | null = null;
  let totalProcessed = 0;

  do {
    // SINGLE execution per page — equivalent to single-threaded getJobs()
    const { workItems, nextCursor } = await getJobsActivity({
      criteria: input.criteria,
      cursor,
      batchSize: input.batchSize
    });

    if (workItems.length === 0) break;

    // PARALLEL execution — equivalent to multi-threaded executeJob()
    // Temporal executes these as concurrent activities on the worker pool
    await Promise.all(
      workItems.map(item => executeJobActivity(item))
    );

    cursor = nextCursor;
    totalProcessed += workItems.length;
  } while (cursor !== null);
}

// activities.ts
export async function getJobsActivity(params: {
  criteria: Record<string, unknown>;
  cursor: string | null;
  batchSize: number;
}): Promise<{ workItems: WorkItem[]; nextCursor: string | null }> {
  // Equivalent to getJobs() — query DB, return page of work items
  const rows = await db.query(
    `SELECT * FROM custom_table WHERE ... AND ($1::text IS NULL OR id > $1)
     ORDER BY id ASC LIMIT $2`,
    [params.cursor, params.batchSize]
  );
  return {
    workItems: rows.map(toWorkItem),
    nextCursor: rows.length === params.batchSize ? rows.at(-1)!.id : null
  };
}

export async function executeJobActivity(item: WorkItem): Promise<void> {
  // Equivalent to executeJob(Document) — process one record
  await deleteRecord(item.id);
}

Custom Framework — Architecture Decision Records

ADR-CF-001 — Kafka over Database-backed Queue for Task Q Equivalent Accepted +

Sterling uses YFS_TASK_Q (a DB table) as its work queue. In a custom framework, we must choose between replicating this pattern (a DB-backed outbox/queue table) or using a dedicated message broker.

Use Kafka as the work queue for Task Q equivalent agents. Domain services publish events to Kafka topics instead of inserting into a queue table. Consumer group workers process events with consumer offset as the state tracking mechanism.

✅ Replay capability — reprocess any historical event window without special tooling. ✅ Fan-out — multiple consumer groups can react to the same event independently. ✅ Scales to millions of events/sec. ✅ Consumer lag is a first-class, scrapable metric for autoscaling. ⚠️ Higher operational complexity than a DB table — Kafka cluster requires dedicated ops. ⚠️ Ordering guarantee only within a partition — must partition by orderId carefully.

DB outbox table (Transactional Outbox pattern): Simpler operationally and provides exactly-once with the source DB transaction, but does not scale beyond ~10K events/sec and lacks replay/fan-out. Good choice for low-volume deployments or when Kafka is unavailable. Redis Streams: Good for simpler fan-out and lower latency than Kafka, but limited retention and weaker durability guarantees for mission-critical order events.

ADR-CF-002 — Temporal.io for Non-Task Q and Complex Time-Triggered Agents Accepted +

Non-Task Q agents have complex multi-step execution: fetch batch → parallel process → fetch next batch → repeat. The Sterling implementation uses JMS + YFS_OBJECT_LOCK to manage state. In a custom framework, managing this state reliably across restarts is non-trivial.

Use Temporal.io for Non-Task Q agents and any time-triggered agent with multi-step logic. Temporal's durable execution model makes getJobsActivity + executeJobActivity resumable across worker pod restarts with zero custom state management code. Use K8s CronJobs only for simple single-step batch jobs.

✅ Workflow history survives pod restarts — no equivalent to the "stuck INPROCESS" problem in Sterling. ✅ Per-activity retry without re-running prior steps. ✅ Temporal UI provides Sterling-equivalent visibility into batch job state. ✅ Workflow deduplication replaces YFS_OBJECT_LOCK. ⚠️ Temporal cluster adds operational overhead — requires Temporal Server (or Temporal Cloud). ⚠️ Learning curve for team unfamiliar with Temporal's programming model.

Custom state machine in Redis: Achievable but requires significant custom code for pagination cursor, retry logic, and distributed locking — essentially rebuilding what Temporal provides. Apache Airflow: Good for data pipelines but Python-centric and not designed for high-frequency, low-latency operational workflows. K8s CronJob alone: Insufficient for multi-step stateful workflows — job restart loses cursor position.

ADR-CF-003 — KEDA for Event-Driven Autoscaling of Consumer Workers Accepted +

Sterling scales agents by increasing MaxThreads and adding JVM instances — a manual, reactive process. In a Kubernetes-native framework, we want agents to scale automatically and proactively based on actual queue depth, not CPU/memory lag indicators.

Use KEDA (Kubernetes Event-Driven Autoscaling) with Kafka trigger for Task Q workers and Integration Server workers. KEDA natively scrapes Kafka consumer group lag and scales pod count to maintain a target lag-per-replica ratio. For Temporal workers, use KEDA's Temporal scaler targeting task queue depth.

✅ Scale-to-zero during off-peak — cost savings vs always-on Sterling JVMs. ✅ Proactive scale-out before lag causes latency — reacts within 30s vs Sterling's manual tuning. ✅ Declarative config in K8s YAML — version-controlled, reviewable. ⚠️ KEDA must be installed as a cluster add-on. ⚠️ Scale-down must be tuned carefully — aggressive scale-down with in-flight Kafka messages can cause rebalance latency.

CPU-based HPA: Lags 2–3 minutes behind actual queue depth growth — unacceptable for order processing SLA during flash sales. Custom metric HPA (Prometheus adapter): Viable but requires more configuration than KEDA's native Kafka scaler. KEDA is the de-facto standard for this use case.

ADR-CF-004 — Redis for Distributed Idempotency Keys Across All Agent Types Accepted +

At-least-once delivery (Kafka, Temporal activities) means the same work item can be processed more than once. Sterling's YFS_TASK_Q STATUS column provided a basic idempotency guard. A custom framework needs an equivalent that works across stateless pods and multiple agent types.

Use Redis SET NX with TTL as the shared idempotency store for Task Q workers and Integration Server workers. Key = deterministic hash of event identity (topic:partition:offset or correlationId). TTL = 24h. For Temporal, rely on built-in workflow/activity deduplication — no Redis needed.

✅ Sub-millisecond idempotency check — no DB round trip. ✅ TTL-based auto-expiry — no housekeeping needed. ✅ Works across pods — Redis is shared infrastructure. ⚠️ Redis failure = idempotency checks skipped — mitigate with Redis Sentinel or Cluster. ⚠️ Key eviction under memory pressure could allow re-processing — set maxmemory-policy: noeviction for the idempotency key namespace.

DB-backed idempotency table: Correct but adds DB write on every message — at high throughput this becomes a bottleneck. Acceptable for <1K messages/sec. In-process deduplication (Bloom filter): Does not survive pod restart and does not work across multiple pods.

ADR-CF-005 — Kafka Connect for External System Integration (replacing Sterling IS bridge) Accepted +

Sterling's Integration Server is tightly coupled to JMS/MQ and requires custom Java handlers for every message type. In a modern integration layer, we want managed connectors, schema validation, and no custom Java per integration point.

Use Kafka Connect with managed connectors (IBM MQ Source, HTTP Source, CDC Source via Debezium) for protocol bridging. Use Confluent Schema Registry + Avro for schema evolution governance. Custom Router Service pods handle routing and transformation — thin, stateless, independently deployable per integration domain.

✅ New external system integration = new Kafka Connect connector config — no code change to router. ✅ Schema Registry prevents breaking changes from source systems silently corrupting data. ✅ Router Service is independently deployable — update transform logic without touching connectors. ⚠️ Kafka Connect cluster adds operational overhead. ⚠️ Connector availability depends on Confluent Hub / community — bespoke legacy protocols may need custom connectors.

Apache Camel: Feature-rich integration framework but heavyweight; adds significant code surface area. Good choice if team already has Camel expertise. MuleSoft / Boomi: Managed iPaaS — reduces ops burden but high licensing cost and vendor lock-in; unsuitable for GCC cost center model.
